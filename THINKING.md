# Market Radar — 프로덕트 의사결정 기록

> 이 문서는 투자 시세 모니터링 대시보드 **Market Radar**를 기획·운영하며
> 내가 어떤 문제를 정의하고, 어떤 가설을 세우고, 왜 그 판단을 내렸는지를 기록한다.
>
> **역할**: Product Owner (기획·데이터 판단·UX 방향 결정 전담)
> **기간**: 2026.03 ~
> **제품**: Market Radar — 국내주식·미국주식·암호화폐 통합 실시간 대시보드
>
> 결과보다 **사고 과정**을 중심으로 기록한다.
> 무엇을 만들었는가보다, 왜 그 방향을 선택했는가 — 이 질문에 답하는 문서다.

---

## 작성 원칙

- **관찰 → 정의 → 가설 → 판단** 순서로 기록한다
- 사용자 여정의 어느 지점에서 문제가 발생했는지 명시한다
- 선택지가 여러 개였다면 왜 그중 하나를 골랐는지 근거를 남긴다
- "됐습니다" 보고를 그냥 넘기지 않고, 실제로 확인했는지 기록한다
- 틀렸을 때도 기록한다 — 왜 그 판단이 틀렸는지가 더 중요한 학습이다

(일반화 가능한 원칙 형태로)

## Case 1. "지수를 못 받아오는 건 니가 문제 아니야?"

### 내가 관찰한 것

개발팀에 "시장 지수가 누락된다"고 보고했더니, 돌아온 답은 "Vercel 환경변수가 설정되었는지 확인해달라"였다. 이미 설정되어 있다고 여러 번 말했는데도 같은 질문이 반복됐다.

### 왜 이게 문제라고 느꼈는가

개발팀이 문제의 원인을 외부(환경 설정)에서 찾고 있었다. 하지만 나는 실제로 앱을 쓰면서 "어떤 지수는 나오고 어떤 건 안 나온다"는 패턴을 봤다. 설정 문제라면 전부 안 나와야 정상이다. **일부만 실패하는 건 코드 로직 문제다.**

### 내 판단

> "버셀에 키가 있대도..? 왜 자꾸 물어보는거야"
> "지수를 못 받아오는 니가 문제 아니야? 개발팀 QA 이거 원인파악해 그리고 CTO 영입해 그냥"

단순한 버그 리포트가 아니라, **시스템 전체를 감사해야 한다**고 판단했다. 이유:
- 같은 종류의 데이터 이슈가 반복 발생하고 있었다 (등락률 0%, 지수 누락, sparkline 빈칸)
- 개별 fix로는 다른 곳에서 또 터질 것 같았다
- "CTO급 전수 감사"를 요청한 건, 증상이 아니라 구조를 봐달라는 의미였다

### 결과

감사 결과 실제로 코드 문제였다:
- NDX 지수가 `^IXIC`(나스닥 종합, 18,000)을 호출하면서 UI에는 "NDX"(나스닥-100, 21,000)로 표시
- 등락률 비교 함수가 4개 파일에 중복되어 있고, 모두 같은 방식으로 잘못됨
- SSRF 보안 취약점 3건 발견

### 내가 배운 것

**개발팀이 "환경 설정 확인해달라"고 할 때, 그건 코드 안에서 원인을 못 찾았다는 신호일 수 있다.** PO로서 "그건 이미 했다"고 명확히 끊고, 조사 방향을 전환시키는 것도 역할의 일부다. 그리고 반복되는 문제에는 "한 번 더 고쳐줘"가 아니라 "전체를 봐"라고 에스컬레이션하는 판단이 필요하다.

## Case 2. "비트 뉴스면 비트만 보여주면 안 되지"

### 내가 관찰한 것

비트코인 관련 뉴스를 클릭했을 때, 관련 종목에 BTC 하나만 표시됐다. 재건축 수주 뉴스를 클릭했을 때 건설사가 하나도 안 나왔다.

### 왜 이게 문제라고 느꼈는가

투자자가 "비트코인 급등" 뉴스를 보면 머릿속에 떠오르는 건 BTC만이 아니다:

> "비트 관련 뉴스면 비트 말고 메이저코인 몇 개나 스트래티지(MSTR), BTC ETF 같은 건 보여야지? 코인베이스나..? 재건축 수주 뉴스에 건설사 하나도 없는 것도 이상하고. 무슨 말인지 알지?"

나는 투자자로서의 사고 흐름을 알고 있다. BTC 뉴스를 보면 자연스럽게 "IBIT는 어때?", "MSTR은?", "ETH도 같이 올랐나?"를 확인한다. 이 연쇄적 탐색이 제품 안에서 끊기고 있었다.

### 내 판단

문제는 "검색 알고리즘이 약하다"가 아니라, **투자자의 멘탈 모델과 제품의 데이터 연결 구조가 불일치**한 것이었다.

확인해보니 코드에 `relatedAssets.js`라는 948줄짜리 연관 자산 매핑 파일이 이미 있었다. BTC → IBIT, MSTR, COIN, ETH 등 15개 관계가 정의되어 있었다. **하지만 뉴스 매칭에서는 이걸 전혀 안 쓰고 있었다.** 차트 상세 패널에서만 사용 중이었다.

### 해결 방향

"새로운 관계 데이터를 만들어라"가 아니라, **이미 있는 데이터를 올바른 곳에 연결하라**.
- 뉴스 → 키워드 직접 매칭(기존) → 매칭된 종목의 연관자산 자동 확장(추가)
- BTC 매칭 → IBIT, MSTR, COIN, ETH 등 최대 8개까지 표시

### 내가 배운 것

**기능이 없는 게 아니라 연결이 안 되어 있을 때가 더 아깝다.** 관계 데이터 948줄을 이미 공들여 만들어놓고 한 곳에서만 쓰고 있었다. PO가 데이터 흐름 전체를 보지 않으면 이런 "자원 낭비"를 놓치게 된다.

## Case 3. "휴장이면 왜 종목이 아예 안 보여?"

### 내가 관찰한 것

국내 장이 닫히면 "국내 급등" 섹션에 "🌙 장마감"이라는 오버레이만 뜨고, 종목 리스트가 완전히 사라졌다.

### 왜 이게 문제라고 느꼈는가

> "국장/미장 급등급락 '휴장' 표시까진 좋은데 왜 종목들이 아예 안 보여? 딤 처리되고 장마감 기준으로는 있어야 되는 거 아니야?"

나는 **장이 닫혀도 "어제 뭐가 올랐지?"를 확인하고 싶다.** 다음 날 장 시작 전에 어제의 급등/급락을 복기하는 건 트레이더의 기본 루틴이다. 그런데 이 데이터가 있는데도 "휴장"이라는 이유로 통째로 숨기고 있었다.

### 내 판단

- "데이터가 없어서 못 보여주는 것"과 "데이터가 있는데 숨기는 것"은 완전히 다르다
- 휴장 상태 표시는 필요하지만, 그 방법이 "전체 숨김"이어서는 안 된다
- **딤 처리(흐리게) + 작은 라벨**이면 "이건 실시간이 아니다"를 전달하면서 데이터는 유지할 수 있다

### 해결 방향

오버레이(종목 전체 대체) → 딤 처리(opacity 60%) + 인라인 라벨("🌙 장마감 · 전일 종가 기준")
- 4개 섹션(국내 급등/급락, 미장 급등/급락) 모두 동일하게 적용
- 코인은 24시간이므로 변경 없음

### 내가 배운 것

**"없음"을 표현하는 방식이 UX의 수준을 결정한다.** 빈 화면, 로딩 스피너, 에러 메시지, 딤 처리 — 각각이 사용자에게 전달하는 의미가 다르다. "데이터가 있는데 실시간이 아닌 상태"에는 "흐리게 보여주기"가 맞다.

## Case 4. "뉴스 눌렀는데 왜 밖으로 나가?"

### 내가 관찰한 것

뉴스 항목을 클릭하면 외부 기사 링크로 바로 이동했다. 사용자가 앱을 떠나버렸다.

### 왜 이게 문제라고 느꼈는가

> "뉴스는 간략하게 요약, 그리고 관련 종목들을 노출시키는 방향으로. 종목상세처럼 스윽 나오게"

투자자가 뉴스를 클릭하는 이유는 "기사를 정독"하기 위해서가 아니다. **"이 뉴스가 내 포트폴리오에 영향을 주는가?"**를 빠르게 판단하기 위해서다. 외부 이동은 이 판단 흐름을 끊는다.

### 내 판단

뉴스 클릭 시 인앱 사이드 패널을 열어서:
1. AI가 기사를 3문장으로 요약
2. 관련 종목 실시간 가격 표시
3. 같은 주제의 관련 뉴스 추가 노출
4. 원문이 필요하면 하단 "원문 보기" 링크

기존 차트 사이드 패널과 동일한 슬라이드 인터랙션을 쓰면 사용자가 이미 익숙한 패턴이다.

### 내가 배운 것

**사용자가 앱을 떠나는 순간 = 서비스가 가치를 제공할 기회를 잃는 순간.** 외부 링크가 아니라 인앱 경험으로 전환하는 건 단순 UI 변경이 아니라 "사용자의 판단 흐름을 우리 안에서 완결시키겠다"는 전략적 결정이다.

## Case 5. "제목을 따라가버리네?"

### 내가 관찰한 것

AI 뉴스 요약 기능을 달았는데, 결과물이 기사 제목을 반복하는 수준이었다.

### 왜 이게 문제라고 느꼈는가

> "뉴스 요약이 제대로 동작하지 않는 것 같아. 기사 글 전문이 몇 줄로 깔끔하게 요약되어야 하는데, 그냥 제목을 따라가버리네?"

제목만 반복하면 요약이 아니다. "AI 요약"이라는 라벨을 달아놓고 제목을 앵무새처럼 반복하면 **사용자가 AI 기능 전체를 신뢰하지 않게 된다.**

### 내 판단

원인을 추적해보니, RSS에서 받은 `description` 필드(보통 1~2문장)만 AI에 넘기고 있었다. **기사 본문을 크롤링하지 않고 있었던 것.** 입력이 제목+1문장이면 출력도 그 수준을 벗어날 수 없다.

해결 방향:
1. 기사 URL로 HTML 크롤 → `<article>` 태그 또는 `<p>` 태그에서 본문 추출
2. 크롤 실패 시 제목+RSS description 조합으로 fallback
3. 프롬프트도 분리: 본문이 있으면 "핵심 3문장 요약", 없으면 "이 정보로 투자 시사점 분석"

### 내가 배운 것

**AI 기능의 품질은 모델이 아니라 입력 데이터가 결정한다.** GPT-4든 Gemini든, 제목 1줄 넣으면 제목 수준의 답이 나온다. "AI 요약이 별로야"라는 피드백의 80%는 프롬프트나 모델이 아니라 **데이터 파이프라인** 문제다.

## Case 6. "무료 모델 없어?"

### 내가 관찰한 것

AI 요약 기능을 위해 API 키가 필요했다. 유료 모델 목록을 보여줬더니 바로 질문이 나왔다.

### 내 사고 과정

> "무료 모델 없어?"

단순히 돈을 아끼려는 게 아니었다. 이 서비스는 **개인 프로젝트**다. 유료 API에 의존하면:
- 매월 고정 비용 발생 → 서비스 유지 부담
- 사용량 증가 시 비용 예측 불가
- 키 관리 포인트 증가

Gemini Flash의 무료 티어(분당 15회, 일 1,500회)를 발견하고, 이 서비스의 뉴스 요약 호출량(하루 50~100회 추정)과 대조해서 무료로 충분하다고 판단했다.

### 내가 배운 것

**비용 구조를 초기에 결정하는 것은 기술 선택이 아니라 비즈니스 판단이다.** "일단 유료로 가고 나중에 최적화"는 개인 프로젝트에서 서비스 종료 사유가 된다. 무료 티어의 한계를 정확히 계산하고, 그 안에서 동작하도록 설계하는 것이 지속 가능한 서비스의 조건이다.

## Case 7. "Gemini 키 rate limit인데 요약은 왜 잘 되지?"

**핵심 판단 한 줄**: 에러 위치(GitHub Actions)와 실제 위험 위치(프로덕션 뉴스)가 달랐다 — 캐시가 만료되면 서비스가 같이 죽는 구조임을 먼저 읽어야 했다.

### 문제 인식

GitHub Actions에서 "Gemini API rate limit exceeded" 에러가 발생했다. 그런데 프로덕션에서 뉴스 요약 기능(같은 API 키)은 정상 동작 중이었다.

> "Gemini API 키 rate limit 걸렸다는데 뉴스 눌러보면 요약 잘 되고 있는데, API 키에 무슨 일이 있는지 확인해줄래?"

### 사용자/서비스 관점

뉴스 AI 요약은 이 서비스의 핵심 차별점이다. "지금은 잘 된다"는 말을 그냥 받아들이면 — CDN 캐시가 만료되는 순간 뉴스 요약이 전면 중단된다. 사용자는 "아까는 됐는데?"라는 최악의 경험을 겪는다. 사용자가 느끼는 신뢰 손상은 기능 일시 중단보다 "불규칙한 중단"이 훨씬 크다.

### 사고 과정

같은 API 키가 rate limit에 걸렸는데 다른 곳에서는 잘 된다. 논리적으로 둘 중 하나다:
1. **키가 다르다** — 확인해보니 동일 키
2. **같은 키인데 하나가 캐시로 버티고 있다** → 이쪽이 가설

조사 결과: 같은 API 키를 두 곳에서 공유하고 있었다.
- 프로덕션 뉴스 요약 (Vercel Edge Function)
- PR 리뷰 봇 Qodo (GitHub Actions)

PR을 연속 3개 생성하면서 Qodo가 API를 과도 호출 → rate limit 도달. 프로덕션 뉴스는 1시간 CDN 캐시가 있어서 겉으로는 정상처럼 보였다.

트레이드오프 검토:
- **A안: 키를 분리** — Qodo용 별도 키 발급. 비용·관리 포인트 증가
- **B안: Qodo 제거** — 리뷰어 4명 → 3명. CI 품질이 다소 낮아질 수 있음
- **판단 기준**: 프로덕션 뉴스 요약 > CI 코드 리뷰 도구 (핵심 기능 우선)

### 내 판단

Qodo를 제거했다. 프로덕션 서비스와 CI 도구가 동일 API 키를 공유하는 구조 자체가 위험하고, Qodo의 가치보다 뉴스 요약 안정성이 중요하다고 판단했다. Gemini + CodeRabbit + Copilot 3명으로도 리뷰 품질은 유지 가능하다.

### 결과

Qodo 제거 후 API rate limit 에러 재발 없음. CDN 캐시 만료 후에도 뉴스 요약 정상 동작.

### PM으로서 배운 것

**"에러가 난 곳"과 "에러의 원인"과 "진짜 위험"은 모두 다를 수 있다.** rate limit 에러는 GitHub Actions에서 났지만, 진짜 위험은 프로덕션이었다. "지금 잘 되고 있으니 괜찮다"는 말을 캐시라는 맥락 없이 받아들이면 시한폭탄을 방치하는 것이다. 시스템의 현재 상태가 아니라 **그 상태가 유지되는 조건**을 보는 습관이 필요하다.

## Case 8. "카톡으로 계속 토큰 발급 알림이 와"

### 내가 관찰한 것

한국투자증권 API를 쓰는데, 내 핸드폰에 카카오톡 알림이 끊임없이 왔다. "API 접근 토큰이 발급되었습니다."

### 왜 이게 문제라고 느꼈는가

> "지금 한국투자증권 API 호출할 때마다 토큰을 매번 새로 발급받아서 내 핸드폰에 카톡 알림 테러가 오고 있어."

서비스의 기술적 문제가 **내 일상생활을 방해**하고 있었다. 30초마다 가격을 polling하니까 30초마다 토큰 발급 → 30초마다 카톡 알림. 토큰 유효기간이 24시간인데 매번 새로 발급받을 이유가 없다.

### 내 판단

> "토큰 유효기간(24시간) 동안은 재사용할 수 있도록 토큰 캐싱 로직을 추가해 줘. 만료됐을 때만 재발급받도록 수정해 줘."

문제 정의부터 해결 방향까지 직접 제시했다. 이건 개발팀에게 "알아서 해줘"가 아니라, 어떻게 고쳐야 하는지 구체적으로 말해야 빠르게 해결된다고 판단했다.

### 후속

수정 후에도 15분 뒤 다시 알림이 왔다. 캐싱이 Edge Function 특성상 인스턴스 간 공유가 안 되는 문제였다. **1차 수정이 됐다고 끝이 아니라, 실제로 알림이 멈췄는지 검증해야 한다.** 이후 Serverless 런타임으로 전환해서 해결.

### 내가 배운 것

**사용자 경험은 화면 안에만 있지 않다.** API 호출 한 번이 사용자의 카톡 알림으로 이어질 수 있다. 기술적 의사결정의 영향 범위를 "우리 앱 안"으로 한정하면 놓치는 것들이 있다.

## Case 9. "한투 API가 있는데 왜 전종목을 못 보여줘?"

### 내가 관찰한 것

한국투자증권 API 키를 세팅했고, 일부 종목은 잘 나온다. 그런데 검색해서 추가한 종목은 가격이 안 나온다. 코스피·코스닥 전종목 + ETF를 커버하지 못하고 있었다.

### 내 사고 과정

> "한국투자증권 API가 있는데도 코스피, 코스닥, 국장에 있는 ETF의 전종목과 가격을 반영하지 못하는 이유 분석해줘"

확인해보니 **코드에 종목 52개가 하드코딩**되어 있었다. 새 종목을 검색해서 watchlist에 추가해도, polling 대상은 항상 그 52개뿐이었다. API는 있는데 코드가 활용하지 않고 있었다.

> "어떻게 해야 전종목을 (미장 포함) 노출할 수 있을까? 한투 아니어도 좋으니 방법을 찾아줘. 유료여도 좋으니"

한투 API에 고정되지 않았다. **목표는 "전종목 커버"이고, 수단은 유연하게** 선택할 수 있다.

### 내 판단

미장 데이터의 경우 "15분 지연"이 아닌 실시간 소스를 원했다:

> "미국 15분 지연 아닌 거 찾아줘"

결과적으로 Yahoo v8(실시간) → Stooq(fallback) → 개별 chart(최종 fallback) 3단 구조로 결정. 국장은 watchlist 기반 동적 polling으로 변경해서, 검색 → 추가 → 가격 표시가 자연스럽게 이어지도록 했다.

### 내가 배운 것

**"API가 있다"와 "API를 활용하고 있다"는 다른 말이다.** 기능이 있는데 안 쓰이고 있으면 그건 기능이 없는 것과 같다. PO는 "이 기능 넣었습니다"를 들었을 때 "근데 실제로 동작해?"를 반드시 확인해야 한다.

## Case 10. "처음 들어오면 아무것도 없는 걸 보게 되잖아?"

### 내가 관찰한 것

관심종목 섹션이 화면의 큰 영역을 차지하는데, 신규 사용자는 아무 종목도 등록하지 않은 상태다. 빈 영역만 크게 보인다.

### 왜 이게 문제라고 느꼈는가

> "관심종목이 너무 빈 채로 크게 영역을 차지하는데, 이 서비스에 처음 들어오면 아무것도 없는 걸 보게 되잖아?"

이건 **첫인상 문제**다. 투자 앱에 처음 들어왔는데 빈 화면이면 "이 앱 콘텐츠가 없나?"라고 느낀다. Cold Start 상태에서의 UX를 설계하지 않은 것이다.

### 내 판단

- 관심종목 영역 크기를 줄여야 한다 (3번 반복 지적한 이슈)
- 빈 상태에서는 "인기 종목 추천" 같은 온보딩 콘텐츠를 보여줘야 한다
- 또는 기본 종목을 미리 넣어두고 사용자가 커스텀하게 해야 한다

### 내가 배운 것

**서비스의 첫 화면은 "가장 많은 데이터가 있을 때"가 아니라 "아무 데이터도 없을 때"를 기준으로 설계해야 한다.** 기존 사용자 기준으로만 보면 Cold Start 문제를 놓치고, 신규 사용자 이탈률이 높아진다.

## Case 11. "나라면 핫한 거 먼저 궁금할 것 같고"

### 내가 관찰한 것

홈 화면에 6개 섹션이 동일한 크기와 무게로 나열되어 있었다. 어떤 섹션이 더 중요한지 시각적 위계가 없었다.

### 내 사고 과정

> "이 서비스 처음 오면 나라면 주목할 만한 (결국 핫한 거) 궁금할 것 같고, 관심종목과 투자 시그널은 훑어볼 것 같아. 아무래도 주목할 만한 - 관심종목 이 2가지에 포커스가 갈 것 같고"

나 자신을 사용자로 놓고 **행동 시퀀스를 시뮬레이션**했다:
1. 앱 열면 → "오늘 뭐가 핫하지?" (주목할 만한)
2. 그다음 → "내 종목은?" (관심종목)
3. 시간 되면 → "무슨 뉴스?" (투자 시그널)

이 순서가 곧 레이아웃 우선순위여야 한다.

### 내 판단

> "급등/급락 6카드 섹션이 살짝 첫 접속 화면에서 걸쳐 보여야 좋은 듯?"

Above-the-fold 전략: 첫 화면에서 핫 종목 카드가 "살짝 걸쳐" 보이면 스크롤 동기가 생긴다. 완전히 보이거나 완전히 안 보이는 것보다, **일부만 보이는 게** 탐색 욕구를 자극한다.

> "투자 시그널 뉴스는 다른 타이틀명과 뉴스 선별도 좀 더 알고리즘 강화하고, 노출 방식도 단순 리스트 말고 뭔가 비주얼화하면 좋을 것 같아. (종목이랑 연계하거나?)"

뉴스를 단순 리스트가 아닌, 종목과 연결된 비주얼 카드로 바꾸자는 제안.

### 내가 배운 것

**레이아웃은 "예쁘게 배치"가 아니라 "사용자의 행동 순서를 반영"하는 것이다.** 내가 이 앱을 열 때 뭘 먼저 보는지를 관찰하면, 어떤 섹션이 위에 와야 하는지가 자연스럽게 결정된다.

## Case 12. "뉴스나왔지만 미반응? 용도를 모르겠음"

### 내가 관찰한 것

홈 화면에 "뉴스가 나왔지만 주가 미반응" 섹션이 있었다.

### 왜 이게 문제라고 느꼈는가

> "뉴스나왔지만 미반응? 그 섹션 제거. 용도를 모르겠음. 핵심뉴스와 알고리즘 검토해서 핵심뉴스 쪽으로 병합"

이 섹션이 **어떤 투자 판단에 도움이 되는지 설명할 수 없었다.** "뉴스가 나왔는데 주가가 안 움직였다"는 정보가 투자자에게 주는 액션은 뭐지? 매수? 매도? 관망? 명확하지 않다.

### 내 판단

기능을 만들었다고 유지해야 하는 건 아니다. **"이 기능의 존재 이유를 한 문장으로 설명할 수 없으면 삭제"**가 내 기준이다. 핵심 뉴스 섹션에 해당 알고리즘의 좋은 부분만 병합하고, 독립 섹션은 제거.

### 내가 배운 것

**기능 삭제도 기획이다.** "만들어놓은 거 아까우니까 두자"는 사용자에게 노이즈를 준다. PO의 역할에는 "무엇을 빼는가"도 포함된다. 뺄 수 있는 용기가 제품을 깔끔하게 만든다.

## Case 13. "억대는 되어야 고래지"

### 내가 관찰한 것

"거래소 대량체결 감지" 섹션에서 소액 거래도 "고래 알림"으로 뜨고 있었다.

### 내 사고 과정

> "고래알림의 금액 상향. 억대는 되어야 고래지"

암호화폐 시장에서 "고래"란 최소 억 단위 거래를 하는 플레이어를 말한다. 천만 원대 거래를 "고래"로 분류하면 알림이 범람하고, 정작 진짜 고래 움직임이 노이즈에 묻힌다.

### 내 판단

도메인 전문 지식에 기반한 기준 재정의:
- 기존: 일정 금액 이상이면 모두 알림
- 변경: 억 단위 이상만 "고래"로 분류

기술적으로는 단순히 threshold 숫자 하나를 바꾸는 것이지만, **그 숫자를 정하는 건 코드가 아니라 도메인 이해**다.

### 내가 배운 것

**임계값(threshold) 설정은 기술이 아니라 비즈니스 판단이다.** 개발팀이 "1000만 원으로 할까요?"라고 물으면, 그건 PO가 시장에 대한 이해를 바탕으로 답해야 한다.

## Case 14. "코인 가격 어떻게 받아오고 있어? 이미지도 종종 안 돼"

### 내가 관찰한 것

코인 가격이 간헐적으로 안 나오고, 코인 로고 이미지도 깨지는 경우가 있었다.

### 내 사고 과정

> "코인 관련정보 잘 못 받아오는 것 같은데, 지금 가격 어떻게 받아오고 있어?"

먼저 **현재 구조를 파악**하려 했다. 문제를 고치기 전에 "지금 어떻게 되어 있는지"를 이해하는 것이 순서다.

> "CoinGecko나 CMC는 API 키 같은 거 없어? 혹은 그런 서비스 무료인데 좋은 거 없어? Web3 쪽은 꽤 있을 것 같아서"

무료 대안을 적극적으로 탐색했다. Web3 생태계의 특성상 오픈 API가 많을 거라는 가설.

> "일단 내가 외부라서 키 발급받고 그런 부분은 어렵고, 바이낸스나 CoinPaprika 이런 거 감안해서 가장 최적화된 (성능 위주, 안정성 위주) 걸 구현해줘"

제약 조건(키 발급 어려움)을 명확히 하고, 그 안에서 **성능과 안정성 기준**으로 판단을 위임했다.

### 내가 배운 것

**제약 조건을 먼저 명확히 하면 선택지가 좁아지고 결정이 빨라진다.** "최고의 코인 API를 찾아줘"가 아니라 "키 발급 없이 + 성능 위주 + 안정성 위주"로 조건을 걸면, 답이 바이낸스 WebSocket + CoinPaprika fallback으로 수렴한다.

## Case 15. "이유분석중만 계속 나와"

### 내가 관찰한 것

"이유 있는 움직임" 카드에 "이유분석중..."이라는 로딩 텍스트만 영구적으로 표시되고, 실제 이유는 나오지 않았다.

### 왜 이게 문제라고 느꼈는가

> "종목 보이는 카드에 이유분석중이라고 써있는데, 뭔가 노출 정보 없이 계속 이유분석중이야"

"분석 중"이라는 말은 **곧 결과가 나온다는 약속**이다. 하지만 결과가 영원히 안 나오면, 그건 거짓 약속이 된다. 사용자 신뢰를 깎는 가장 나쁜 패턴.

### 내 판단

로딩 상태가 일정 시간 이상 지속되면:
- "분석 중" → "데이터 확인 불가" 또는 해당 카드를 아예 숨기기
- **데이터가 없으면 "없다"고 솔직하게 표시**하는 것이 "곧 나옵니다"라고 거짓말하는 것보다 낫다

### 내가 배운 것

**로딩 상태에는 타임아웃이 필수다.** "분석 중"을 무한히 보여주는 건, 빈 화면보다 사용자 경험이 나쁘다. 기대를 만들어놓고 충족하지 않기 때문이다.

---

## Case 16. "섹터로테이션이 조금은 보여야 펼치기를 해볼 것 같아"

### 내가 관찰한 것

섹터 로테이션 섹션을 접힘(collapse) 처리했는데, 접힌 상태에서는 존재 자체를 모른다.

### 내 사고 과정

> "섹터로테이션이 조금은 보여야 펼치기를 해볼 것 같아"
> "섹터 로테이션 같은 건 차라리 별도 메뉴로 빼고, 핫/콜드 일부만 노출해서 별도 메뉴로 유도하면 좋잖아?"

완전히 숨기면 → 아무도 안 씀 → 없는 기능.
전부 보여주면 → 홈이 너무 길어짐.
**일부만 노출(핫/콜드 2~3개) + "더보기"** → 관심 유발 + 깊은 탐색 유도.

### 내 판단

이건 **정보 계층(Information Hierarchy)** 문제다:
- 1수준: 항상 노출 (마켓펄스, 관심종목)
- 2수준: 요약만 노출, 펼침 가능 (섹터 로테이션 핫/콜드)
- 3수준: 별도 페이지 (전체 섹터 상세)

홈 화면의 역할은 "모든 것을 보여주기"가 아니라 **"더 보고 싶게 만들기"**다.

### 내가 배운 것

**"접힘"과 "미리보기"는 다르다.** 접힌 상태 = 숨김 = 발견 불가. 미리보기 = 맛보기 = 탐색 동기 부여. UX에서 "숨기되 존재감은 유지"하는 건 세심한 설계가 필요하다.

## Case 17. "왜 내가 피드백한 것들 다 반영 안 됐어?"

### 내가 관찰한 것

여러 피드백을 줬는데, 배포된 버전에 관심종목 섹션 크기, 섹터 노출 방식 등이 반영되지 않았다.

### 왜 이게 문제라고 느꼈는가

> "배포된 거 맞아? 왜 내가 피드백했던 것들 다 반영 안 됐어? 예를 들어 관심종목 섹션 크기, 섹터 노출 방식, 등"

피드백을 주는 건 시간과 에너지가 든다. 그걸 했는데 반영이 안 되면 **"내 피드백이 무시됐다"는 느낌**을 받게 된다. 이건 제품 문제가 아니라 **실행력 관리** 문제다.

### 내 판단

개발팀이 피드백을 놓친 건지, 우선순위에서 밀린 건지 구분해야 한다:
- 놓친 거라면 → 피드백 트래킹 체계가 필요
- 우선순위에서 밀린 거라면 → 그 판단 근거를 공유해야 함

**PO의 피드백은 "참고 사항"이 아니라 "실행 항목"이다.** 반영하지 않을 거면 왜 안 하는지 커뮤니케이션이라도 있어야 한다.

### 내가 배운 것

**피드백 → 실행 사이의 갭을 관리하는 것도 PO의 일이다.** 피드백을 주고 "알아서 되겠지" 하면 안 된다. 트래킹하고, 확인하고, 배포 전에 체크리스트로 검증해야 한다.

## Case 18. "다크모드 필요성은 잘 모르겠어"

### 내가 관찰한 것

스프린트 계획에 "다크모드 지원"이 포함되어 있었다.

### 내 판단

> "일단 니 판단으로 진행. 다크모드 필요성은 잘 모르겠어"

다크모드는 "있으면 좋은" 기능이지 "없으면 서비스를 못 쓰는" 기능이 아니다. 지금 이 서비스의 우선순위는:
1. 데이터 정확성 (changePct=0 같은 버그)
2. 핵심 UX (레이아웃, 정보 계층)
3. 꾸미기 (다크모드, 애니메이션)

3번을 하기 전에 1, 2가 완벽해야 한다. **기능의 "좋음"이 아니라 "지금 필요함"으로 판단해야 한다.**

### 내가 배운 것

**"No"를 말하는 것도 우선순위 결정이다.** 모든 좋은 아이디어를 다 하면 아무것도 제대로 못 한다. "잘 모르겠어"는 "하지 마"의 완곡한 표현이 아니라, **지금은 판단 근거가 부족하니 보류**하겠다는 결정이다.

## Case 19. 사용자 피드백 8건 — "개별 이슈가 아니라 구조 문제다"

**핵심 판단 한 줄**: 8개의 이슈를 개별 fix로 보지 않고 3개 구조적 패턴(정보 밀도 / 컨텍스트 적응 / 관계 매핑)으로 묶어 근본 원인을 먼저 수정했다.

### 문제 인식

직접 앱을 사용하면서 8가지 이슈를 한 번에 발견했다:

1. 마켓펄스 시장지수에 지수 값이 없음 (변동률만 나옴)
2. 주목할 종목 — 휴장 시 코인만 노출되어야 함
3. 거래소 체결 감지 안 됨
4. 관심종목 영역이 너무 큼
5. 휴장 시 국장/미장 급등급락 숨겨야 함
6. 섹터 로테이션 별도 메뉴 분리 필요
7. 뉴스 관련 종목 매칭이 엉뚱함
8. SurgeBanner의 가치 불명확

### 사용자/서비스 관점

이 앱의 포지셔닝은 "매수 판단 직전 5분"이다. 그런데 8개 이슈 중 1, 7번(데이터 오류)은 **사용자의 투자 판단을 직접 왜곡**한다. 잘못된 지수 값으로 "오늘 시장 괜찮네"라고 판단하거나, 뉴스와 무관한 종목이 관련 종목으로 뜨면 — 서비스가 없는 것보다 나쁘다. 신뢰를 떨어뜨리기 때문이다.

### 사고 과정

8개를 나열하고 보니 공통 패턴이 보였다:

| 이슈 번호 | 패턴 | 근본 원인 |
|-----------|------|-----------|
| 1, 4, 6, 8 | 정보 밀도 | 모든 섹션이 같은 크기·무게 → 정보 계층 미설계 |
| 2, 5 | 컨텍스트 적응 | 시장 상태(장 중/휴장)에 따른 콘텐츠 전환 로직 없음 |
| 3, 7 | 관계 매핑 | 데이터 간 연결 미구현 (뉴스↔종목, 체결↔알림) |

**개별로 고치면 8번 fix → 끝. 패턴으로 보면 구조 1개 수정 → 아직 발견되지 않은 같은 패턴의 이슈까지 예방.**

우선순위 프레임:
```
"이 이슈가 사용자의 투자 판단을 왜곡하는가?"
  → Yes → P0 (즉시 수정)
  → No but 불편 → P1 (이번 스프린트)
  → No and 취향 → P2 (백로그)
```

### 내 판단

P0(1, 7번 데이터 오류) → P1(2, 4, 5번 UX) → P2(6, 8번 구조/취향) 순으로 분류하고, P0를 이번 스프린트에 전부 처리하도록 결정했다. 8개를 동시에 진행하면 아무것도 제대로 안 된다는 판단.

### 결과

P0 2건 수정 후 "데이터가 이상하다"는 피드백이 멈췄다. P1은 다음 스프린트에 처리.

### PM으로서 배운 것

**피드백은 "리스트"가 아니라 "패턴"으로 읽어야 한다.** 8개 이슈를 개별로 보면 8번 일해야 한다. 3개 패턴으로 묶으면 구조 수정 1번으로 아직 발견되지 않은 이슈까지 예방할 수 있다. 그리고 우선순위 프레임 없이 이슈를 나열하면, 팀은 가장 눈에 띄는 것부터 한다 — 그게 꼭 가장 중요한 것은 아니다.

## Case 20. "왜 같은 버그가 5번이나 나와?"

**핵심 판단 한 줄**: 3번째 재발에서 "고쳐줘" 대신 "전수 조사부터"를 요청했다 — 개별 fix가 아니라 코드 구조 문제임을 PO가 먼저 읽었다.

### 문제 인식

"등락률이 0%로 표시된다"는 버그가 5번 이상 수정되고 5번 이상 재발했다.

- 1차: "미장 등락률 0이야" → 수정됨 → 해결된 줄 알았음
- 2차: "또 0이야, 이번엔 다른 종목" → 수정됨 → 불안해짐
- 3차: "ETF도 0이야" → **패턴을 인식**

### 사용자/서비스 관점

등락률 0%는 투자자에게 "이 종목은 오늘 변동 없음"으로 읽힌다. 실제로는 올랐거나 내렸는데. 투자 판단의 가장 기본 데이터가 틀리면 — 사용자는 이 앱의 모든 데이터를 의심하게 된다. 데이터 신뢰는 한 번 무너지면 회복이 어렵다.

### 사고 과정

3번째 재발에서 가설을 바꿨다:

- **기존 가설**: "특정 파일에 버그가 있다" → 해당 파일 고치면 끝
- **수정 가설**: "같은 로직이 여러 파일에 복사-붙여넣기 되어 있다" → A 고치면 B에서 터지는 두더지 게임

개발팀은 매번 "해당 파일 수정 완료"라고 보고했다. 하지만 같은 버그가 다른 파일에서 나온다는 건, 로직이 한 곳에 있지 않다는 뜻이다.

> "개별 파일 수정 말고, 이 로직이 어디에 몇 개 있는지 먼저 전수 조사해. 그리고 한 번에 다 고쳐."

### 내 판단

조사 우선. 수정은 그다음. 전수 조사 결과 `us-price.js`, `market-indices.js`, `etf-prices.js`, `stocks.js` 4개 파일에 동일 비교 함수가 중복되어 있었다. 절대값 비교(`0.01 차이`)를 상대값 비교(`0.01%`)로 4개 파일 동시 수정하도록 결정했다.

### 결과

이후 재발 없음. 수정에 걸린 시간: 개별 수정 5번 > 전수 조사 + 일괄 수정 1번.

### PM으로서 배운 것

**"고쳐졌습니다"를 그냥 받아들이면 안 된다.** 같은 버그가 2번 재발하면 "어떻게 고쳤는지"를 물어야 하고, 3번 재발하면 "구조적 원인이 있는지" 전수 조사를 요청해야 한다. PM이 "왜 반복되는가?"를 묻지 않으면 팀은 계속 같은 방식으로 고친다.

## Case 21. 포지셔닝 전환 — "급등 캐치"에서 "매수 판단 5분"으로

**핵심 판단 한 줄**: 기능 스펙이 계속 흔들리는 근본 원인이 포지셔닝 부재라고 진단하고, "급등 캐치"에서 "매수 판단 5분"으로 서비스 존재 이유를 재정의했다.

### 문제 인식

초기 3주간 fix 커밋이 전체의 67%였다. 기능을 추가해도 금방 깨졌다. SurgeBanner(상단 롤링 배너) 하나에 디자인 변경만 4번. "이 기능 넣을까요?"라는 질문이 반복됐고, 매번 답이 달랐다.

### 사용자/서비스 관점

사용자가 이 앱을 열 때 하는 행동을 역추적했다. "급등주를 발견"한 다음 사용자가 실제로 하는 것:
1. "왜 올랐지?" → 검색
2. "관련 뉴스가 뭐지?" → 다른 앱으로 이동
3. "지금 들어가도 되나?" → 스스로 판단

급등 목록 자체는 네이버 증권 인기검색어, 키움 HTS로도 된다. **우리가 줄 수 있는 차별적 가치는 급등 목록이 아니라 판단에 필요한 맥락**이다.

### 사고 과정

가설: **기능 스펙이 흔들리는 건 포지셔닝이 없기 때문이다.** 포지셔닝이 없으면 "이 기능 필요한가?"에 대한 답이 그날그날 달라진다.

포지셔닝 후보 검토:

| 포지셔닝 | 차별성 | 문제 |
|----------|--------|------|
| "급등주 누구보다 빠르게" | 낮음 — 네이버·키움도 됨 | 대체재 多 |
| "국내·미국·코인 통합" | 중간 — 기술적 차별성 | 사용자 니즈 불명확 |
| "매수 판단 직전 5분" | 높음 — 맥락+판단 재료 제공 | 구현 난이도 높음 |

"매수 판단 5분"이 가장 구체적이고, 기능 우선순위 결정 기준으로도 쓸 수 있다.

### 내 판단

포지셔닝을 전환했다:
```
기존: "급상승 종목을 누구보다 빠르게 캐치"
전환: "매수 결정 직전 5분을 함께하는 앱"
```

이 전환이 바꾼 것:
- 뉴스: "부가 기능" → "핵심 무기" (RSS 소스 10개 추가)
- Phase 순서: UX 먼저 → **데이터 신뢰성 먼저**
- 기능 삭제 기준: "매수 판단에 도움이 되는가?" → DEX 핫 토큰(No) 삭제, 이벤트 캘린더(Yes) 유지

### 결과

포지셔닝 전환 이후 "이 기능 넣을까요?" 질문이 거의 사라졌다. 기준이 생기니 팀 내 판단이 빨라졌다.

### PM으로서 배운 것

**포지셔닝이 흔들리면 모든 기능 결정이 흔들린다.** 포지셔닝은 마케팅 문구가 아니라, 일상적인 기능 우선순위 결정의 필터다. "매수 판단에 도움이 되나?"라는 질문 하나로 수십 개의 기능 결정이 3초 안에 끝났다. 개발 속도를 높이는 가장 빠른 방법은 코드가 아니라 명확한 포지셔닝이다.

## Case 22. "Phase 순서가 곧 전략이다"

### 내 판단 과정

구현할 것들이 많았다. 어떤 순서로 할 것인가?

```
선택지 A: UX 먼저 → 예쁘게 만들고 → 데이터 나중에
선택지 B: 데이터 먼저 → 신뢰할 수 있게 만들고 → UX 나중에
```

나는 B를 선택했다.

**이유**: 등락률이 0%로 뜨는 화면에 아무리 예쁜 UI를 입혀도, 사용자가 "이 앱 데이터 맞아?"라고 의심하는 순간 모든 게 끝이다. **투자 앱에서 데이터 신뢰 = 서비스 존재 이유.**

```
Phase 1: 데이터 신뢰성 (기반 공사)
  → changePct=0 근본 수정, 데이터 소스 fallback 강화, 신뢰도 라벨

Phase 2: UX 재구조화 (가치 재정의)
  → 홈 레이아웃 개편, "왜 지금?" 컨텍스트, 정보 계층 설계

Phase 3: 데이터 소스 확장 (차별화)
  → 코인 250+종 확장, 뉴스 RSS 10개, KIS WebSocket
```

### 내가 배운 것

**"무엇을 먼저 하느냐"가 "무엇을 하느냐"보다 중요하다.** Phase 순서를 잘못 잡으면, Phase 2에서 만든 UI가 Phase 1에서 깨진다. "기반 공사 → 구조 → 외장" 순서는 건축에서도 소프트웨어에서도 같다.

## Case 23. "KRX 키도 있는데 이거 폴백용으로 붙여넣을까?"

### 내가 관찰한 것

한투 API만으로는 전종목 커버가 어려운 상황이었다.

### 내 사고 과정

> "맞다 KRX 키도 있는데 이거 폴백용으로 (용도나 상황은 니가 필요한 곳에) 사용하게 붙여넣을까?"

개발팀이 요청하기 전에 **내가 먼저 자원을 제공**한 케이스다. KRX API 키를 가지고 있다는 걸 떠올리고, 이게 fallback으로 쓸 수 있겠다는 판단을 했다. 용도 판단은 개발팀에 위임하되, 자원 자체는 선제적으로 제공.

확인해보니 KRX API에서 기대한 실시간 시세 기능은 없었다. 즉시 목표를 축소:

> "그런 내용 없어, 그냥 종목 리스트 확보로 가자. ETF나"

**"없으면 없는 대로 쓸 수 있는 범위를 빠르게 재정의"**하는 판단.

### 내가 배운 것

**PO가 자원(API 키, 데이터 소스)을 먼저 파악하고 제공하면 개발 속도가 빨라진다.** "키가 필요합니다" → "발급해주세요" → "발급했습니다" → "넣었습니다" 4단계가, 내가 먼저 제공하면 1단계로 줄어든다.

## Case 24. "이 프로젝트는 왜 GitHub에 정보가 제대로 안 되어 있지?"

### 내가 관찰한 것

다른 프로젝트의 GitHub 페이지를 보다가, 우리 레포에는 README, 보안 지침, 프로젝트 설명이 부실하다는 걸 발견했다.

### 왜 이게 문제라고 느꼈는가

> "왜 이 프로젝트는 깃에 이 캡쳐같이 정보 요약이나 README라던지 보안 지침 등이 제대로 안 되어 있지?"

포트폴리오로도 쓸 건데, GitHub 레포가 날것 그대로면 외부에서 볼 때 **"관리가 안 되는 프로젝트"**로 보인다. 코드 품질과 별개로, 프로젝트의 **외부 인식** 관리도 필요하다.

### 내가 배운 것

**코드만 좋아서는 안 된다. 코드를 설명하는 문서도 제품의 일부다.** 채용 담당자가 GitHub을 볼 때 코드를 한 줄 한 줄 읽지 않는다. README, 커밋 히스토리, Issue/PR 관리 상태를 본다.

## Case 25. "QA 돌려주고, 그다음 스프린트 계획 잡고, 나랑 진행은 논의"

### 내 사고 과정

> "QA 돌려주고 그다음 스프린트 진행, 계획 먼저 잡고 나랑 진행은 논의"

이건 **의사결정 프로세스 자체에 대한 결정**이다:
1. QA 먼저 (현재 품질 확인)
2. 계획 수립 (다음 할 일 정리)
3. 나와 논의 (실행 전 합의)

"알아서 해줘"가 아니라 **"계획까지는 니가 해, 실행은 내가 보고 판단할게"**라는 위임의 범위 설정.

### 내가 배운 것

**위임은 "전부 맡기기"가 아니라 "어디까지 맡기는지 경계를 정하는 것"이다.** 조사·계획은 위임하되, 실행 승인은 직접 하는 구조가 가장 효율적이면서 통제력을 유지한다.

## Case 26. "종목상세처럼 스윽 나오게" — 인터랙션 패턴의 일관성

### 내가 관찰한 것

뉴스를 클릭하면 외부 사이트로 이동하는 반면, 종목을 클릭하면 오른쪽에서 차트 패널이 슬라이드로 나타났다.

### 내 사고 과정

> "뉴스는 간략하게 요약, 그리고 관련 종목들을 노출시키는 방향으로. 종목상세처럼 스윽 나오게"

**이미 제품 안에 "좋은 인터랙션 패턴"이 있었다.** 종목 클릭 → 사이드 패널 슬라이드는 사용자가 이미 학습한 패턴이다. 뉴스도 같은 패턴을 쓰면:
- 사용자가 새로운 인터랙션을 배울 필요 없음
- "클릭하면 오른쪽에서 나온다"는 멘탈 모델이 강화됨
- 개발도 기존 컴포넌트 재활용 가능

반대로 뉴스만 "외부 이동"이면, **같은 앱 안에서 클릭의 결과가 불일치**한다. 종목 클릭 = 인앱, 뉴스 클릭 = 외부 이동. 이건 사용자를 혼란스럽게 한다.

### 내 판단

**"같은 제스처(클릭)에 같은 반응(사이드 패널)"**이 UX 일관성의 핵심이다. 새로운 UI를 발명하지 않고, 이미 검증된 패턴을 확장하는 것이 가장 안전하다.

### 내가 배운 것

**좋은 UX는 발명이 아니라 일관성이다.** 제품 안에 이미 잘 동작하는 패턴이 있으면, 그걸 다른 맥락에도 적용하는 것이 사용자 학습 비용을 줄이는 가장 효과적인 방법이다.

## Case 27. "맨 밑에 있으면 있는지도 모를 것 같아" — 콘텐츠 발견성 설계

### 내가 관찰한 것

이벤트 캘린더(FOMC, CPI 발표 등 경제 이벤트)가 홈 화면 맨 아래에 풀 섹션으로 배치되어 있었다.

### 왜 이게 문제라고 느꼈는가

> "이벤트캘린더가 이 서비스 맨 밑에 있으면 있는지도 모를 것 같아. 펄스와 관심종목 사이에 가장 가까운 이벤트 n개가 롤링되고, 누르면 모달로 캘린더나 타임라인 떠야 돼"

**스크롤을 끝까지 내리는 사용자는 거의 없다.** 경제 이벤트는 투자 판단에 중요한 정보인데, 맨 아래에 두면 대부분의 사용자가 존재 자체를 모른다. 이건 "기능이 없는 것"과 실질적으로 같다.

### 내 판단

콘텐츠의 가치와 노출 위치가 불일치할 때의 해결:
- **전체를 상단으로 올리기**: 홈이 길어져서 다른 콘텐츠를 밀어냄 → 거부
- **요약만 상단에 롤링**: 가장 가까운 이벤트 2~3개를 마켓펄스 아래에 한 줄 롤링 → 채택
- **상세는 모달/타임라인**: 롤링 클릭 시 전체 캘린더를 모달로 표시

이건 **"요약-상세 2계층 구조"**다:
```
1계층: 홈 상단 롤링 (발견성 확보)
  → "FOMC 금리 결정 D-3" 한 줄
2계층: 모달 (깊은 탐색)
  → 클릭 시 전체 캘린더/타임라인
```

### 내가 배운 것

**콘텐츠의 가치가 높아도 발견되지 않으면 가치가 0이다.** 중요한 정보일수록 사용자가 "찾아서 봐야 하는" 위치가 아니라 "자연스럽게 눈에 들어오는" 위치에 있어야 한다. 전체를 옮기는 게 아니라 **"미끼(요약)"를 상단에 두고 "본체(상세)"는 별도 진입점**으로 분리하는 게 정보 계층 설계의 핵심이다.

## Case 28. "핵심뉴스 눌러도 뉴스섹션처럼 나와야지" — 동일 콘텐츠 동일 경험

### 내가 관찰한 것

오른쪽 뉴스 탭에서 뉴스를 클릭하면 AI 요약 + 관련 종목이 나오는데, "오늘의 핵심뉴스" 섹션에서 같은 뉴스를 클릭하면 다른 동작을 했다.

### 왜 이게 문제라고 느꼈는가

> "오늘의 핵심뉴스 눌러도 뉴스섹션처럼 요약정보와 관련종목이 나와야지"

**같은 콘텐츠(뉴스)인데 진입 경로에 따라 경험이 다르다.** 사용자는 "뉴스를 클릭했다"고 인식하지, "오른쪽 뉴스탭에서 클릭했는지 홈 핵심뉴스에서 클릭했는지"를 구분하지 않는다. 경로가 다르다는 건 개발의 사정이지, 사용자의 관심사가 아니다.

### 내 판단

**규칙: "같은 유형의 콘텐츠 → 어디서 진입하든 같은 상세 경험"**
- 홈 핵심뉴스 → 뉴스 사이드패널
- 오른쪽 뉴스탭 → 뉴스 사이드패널
- 관련뉴스 목록 → 뉴스 사이드패널

하나의 뉴스 상세 컴포넌트를 만들고, 모든 진입점에서 동일하게 호출.

### 내가 배운 것

**사용자 경험의 일관성은 "같은 콘텐츠에 같은 반응"에서 시작한다.** 개발팀은 "핵심뉴스 섹션"과 "뉴스 탭"을 별개 컴포넌트로 만들었지만, 사용자에게는 둘 다 "뉴스"다. 컴포넌트 구조가 아니라 **사용자의 인식 단위**로 경험을 설계해야 한다.

## Case 29. "지금 뜨는" → "주목할만한" — 네이밍이 곧 콘텐츠 기준

### 내가 관찰한 것

홈에 "지금 뜨는 종목"이라는 큰 섹션이 있었다. 급등/급락 6개 카드가 나열되어 있었는데, 섹션 이름과 내용이 잘 맞지 않았다.

### 내 사고 과정

> "원래 있던 '지금 뜨는?' 어쩌구 큰 섹션을 '주목할만한'으로 변경해서 최대 5개 노출. 이거 관련해서 팀미팅 진행"

**"지금 뜨는"이라는 이름은 기준이 모호하다.** 거래량? 등락률? 뉴스 언급? 뭘 기준으로 "뜨는" 건지 알 수 없다. "주목할만한"으로 바꾸면:
- "왜 주목해야 하는지" 이유를 같이 보여줘야 한다는 콘텐츠 기준이 생긴다
- 최대 5개로 제한해서 "엄선된 느낌"을 줄 수 있다
- 단순 급등 리스트가 아니라 "큐레이션"이 된다

### 내 판단

네이밍 변경은 단순 텍스트 교체가 아니라 **콘텐츠 선별 기준의 재정의**다:
```
"지금 뜨는" → 급등률 순 정렬 (기계적)
"주목할만한" → 급등 + 이유(뉴스/수급) + 컨텍스트 (큐레이션)
```

> "주목할만한 건 별도의 알고리즘을 통해 종목 하나하나 강조해서 해달라는 거였어"

단순 정렬이 아니라, 각 종목이 **왜 지금 주목할만한지** 1문장 설명이 붙어야 한다.

### 내가 배운 것

**섹션 이름을 바꾸면 콘텐츠 기준이 바뀌고, 알고리즘이 바뀌고, 사용자 기대가 바뀐다.** "지금 뜨는"은 "데이터를 나열"하는 섹션이고, "주목할만한"은 "판단을 도와주는" 섹션이다. 네이밍은 UI 텍스트가 아니라 **제품 방향**이다.

## Case 30. "6박스 유지해" — 제거하려는 것을 지킨 이유

### 내가 관찰한 것

개발팀이 홈 리디자인을 하면서 기존 6박스(국내 급등 3 + 국내 급락 3, 미장 급등 3 + 미장 급락 3) 카드 형태를 다른 UI로 교체하려 했다.

### 내 사고 과정

> "원래 있던 6박스 유지하고, 주목할만한 건 별도의 알고리즘을 통해 종목 하나하나 강조해서 해달라는 거였어"

개발팀은 "새 디자인이 더 좋으니까" 기존 6박스를 제거하려 했다. 하지만 나는 반대했다. 이유:
- 6박스는 **한눈에 시장 온도를 파악**할 수 있는 패턴이다 (급등 3개 vs 급락 3개)
- 이미 사용자가 익숙해진 레이아웃을 이유 없이 바꾸면 혼란
- "주목할만한"은 6박스와 **다른 역할**이다 — 6박스는 시장 전체, 주목할만한은 개별 종목 깊이

### 내 판단

**"새것을 만들기 위해 기존 것을 버리지 마라."** 두 가지가 다른 역할을 하면 공존시키는 게 맞다:
```
6박스: 시장 온도 (급등 vs 급락, 한눈에)
주목할만한: 개별 종목 딥다이브 (왜 지금 이 종목인지)
```

### 내가 배운 것

**리디자인은 "전부 새로 만들기"가 아니다.** 기존에 잘 동작하는 것은 유지하고, 부족한 부분만 추가하는 것이 사용자에게 가장 부드러운 전환이다. "있던 걸 없앤다"는 결정은 "없던 걸 만든다"보다 신중해야 한다.

## Case 31. "핵심뉴스와 오른쪽 뉴스탭 차별화" — 같은 콘텐츠, 다른 역할

### 내가 관찰한 것

홈 화면에 "오늘의 핵심뉴스" 섹션이 있고, 오른쪽에 뉴스 탭도 있다. 둘 다 뉴스를 보여주는데, 차이가 뭔지 모호했다.

### 내 사고 과정

> "오늘의 핵심뉴스와 오른쪽 뉴스탭 차별화 고민"

같은 콘텐츠 유형(뉴스)이 두 곳에서 나오는데, 역할이 명확하지 않으면:
- 사용자가 "뭐가 다른 거지?" 혼란
- 개발팀이 "어디에 어떤 뉴스를 넣지?" 혼란
- 결국 두 곳에 비슷한 뉴스가 중복 노출

### 내 판단

같은 콘텐츠라도 **역할을 명확히 나누면** 공존할 수 있다:
```
홈 "핵심뉴스": 에디터 픽 — 가장 임팩트 큰 3~5건, 비주얼 카드형
  → "오늘 이것만은 봐야 한다"

오른쪽 뉴스탭: 전체 피드 — 시간순 전체 뉴스, 리스트형
  → "더 보고 싶으면 여기서 탐색"
```

역할이 다르면 UI도 달라야 한다. 핵심뉴스는 카드, 뉴스탭은 리스트. 같은 리스트 형태면 구분이 안 된다.

### 내가 배운 것

**같은 콘텐츠 유형이 여러 곳에 나올 때는 "이 두 곳의 역할이 다르다"를 UI로 증명해야 한다.** 역할 차이가 시각적으로 드러나지 않으면, 사용자에게는 그냥 중복이다.

## Case 32. "상단 롤링이 제공하고자 하는 가치에 대한 고민이 필요해" — 모든 UI 요소는 존재 이유가 필요하다

### 내가 관찰한 것

화면 최상단에 급등 종목이 좌우로 롤링되는 SurgeBanner가 있었다. 디자인 변경만 4번째.

### 왜 이게 문제라고 느꼈는가

> "상단의 롤링이 제공하고자 하는 가치에 대한 고민이 필요해"

디자인을 4번이나 바꿔야 했다는 건, **이 컴포넌트의 존재 이유가 불명확**하다는 신호다. 역할이 명확하면 디자인은 자연스럽게 따라온다. 역할이 불명확하니까 "이렇게 해볼까? 저렇게 해볼까?"를 반복한 것이다.

### 내 사고 과정

상단 롤링이 줄 수 있는 가치:
1. **속보성** — "지금 이 순간 가장 핫한 것" → 그러면 실시간 갱신 + 이유 표시 필수
2. **분위기** — "오늘 시장이 좋다/나쁘다" → 그러면 지수 변동률이 더 적합
3. **장식** — 화면이 움직이니까 활력 있어 보임 → 그러면 가치가 아니라 노이즈

어떤 역할인지 정하지 않으면, 세 가지가 섞여서 애매해진다.

### 내 판단

존재 이유를 정할 수 없으면 **일단 제거하거나 축소**하고, 다른 것에 집중하는 게 낫다. "있으면 좋겠지"로 유지하는 UI 요소가 쌓이면 화면이 노이즈로 가득 찬다.

### 내가 배운 것

**UI 요소의 존재 이유를 한 문장으로 말할 수 없으면, 그건 없어도 되는 것이다.** "왜 이게 여기 있지?"라는 질문에 3초 안에 답할 수 없는 컴포넌트는 사용자도 무시하고 있을 가능성이 높다.

## Case 33. "뉴스 노출 방식을 단순 리스트 말고 비주얼화" — 콘텐츠와 데이터의 연결

### 내가 관찰한 것

뉴스가 제목만 줄줄이 나열된 리스트 형태였다. 투자 뉴스인데 종목과의 연결고리가 시각적으로 보이지 않았다.

### 내 사고 과정

> "투자시그널 뉴스는 다른 타이틀명과 뉴스 선별도 좀 더 알고리즘 강화하고, 노출 방식도 단순 리스트 말고 뭔가 비주얼화하면 좋을 것 같아. (종목이랑 연계하거나?)"

일반 뉴스앱은 제목 리스트로 충분하다. 하지만 **투자 대시보드의 뉴스는 "이 뉴스가 어떤 종목에 영향을 주는가?"**가 핵심이다. 제목만 보여주면 투자자가 머릿속으로 "이건 어떤 종목에 영향이지?" 연결 작업을 해야 한다.

### 내 판단

뉴스 카드에 관련 종목 뱃지를 직접 붙이는 방식:
```
┌─────────────────────────────────────┐
│ "삼성전자 HBM3E 납품 확대"           │
│ 📈 삼성전자 +2.3%  SK하이닉스 +1.8% │
│ 30분 전 · 한경                       │
└─────────────────────────────────────┘
```

뉴스 제목 아래에 관련 종목의 실시간 등락이 같이 보이면, **"이 뉴스 때문에 이 종목이 움직이고 있구나"**를 한 눈에 파악할 수 있다. 뉴스와 시세 데이터를 **시각적으로 연결**하는 것이 투자 대시보드만의 차별점이다.

### 내가 배운 것

**콘텐츠의 가치는 "무엇을 보여주느냐"뿐 아니라 "무엇과 연결해서 보여주느냐"에 달려있다.** 뉴스 + 종목 가격을 같이 보여주면 정보의 가치가 1+1=3이 된다. 개별 정보를 나열하는 것이 아니라 **정보 간의 관계를 시각화**하는 것이 대시보드의 본질이다.

## Case 34. "걸쳐 보여야" — 스크롤 동기를 설계하다

### 내가 관찰한 것

첫 화면(Above-the-fold)에 마켓펄스와 관심종목만 보이고, 그 아래 콘텐츠는 스크롤해야 보였다.

### 내 사고 과정

> "급등/급락 6카드 섹션이 살짝 첫 접속 화면에서 걸쳐 보여야 좋은 듯?"

**"걸쳐 보인다"는 건 의도적인 UX 기법이다.** 완전히 보이면 "이미 다 봤다"고 느끼고, 완전히 안 보이면 "아래에 뭐가 있는지 모른다." 하지만 **카드의 상단 20%만 걸쳐 보이면** "뭔가 더 있네? 내려볼까?"라는 탐색 동기가 생긴다.

### 내 판단

이건 콘텐츠 순서와 간격의 정밀 조정이다:
```
Viewport (첫 화면)
┌────────────────────────┐
│ 마켓펄스 (지수/환율)    │  ← 완전 노출
│ 이벤트 롤링             │  ← 완전 노출
│ 관심종목 (3~4행)        │  ← 완전 노출
│ ─── 급등/급락 카드 ──── │  ← 상단 20%만 걸침
└────────────────────────┘
  ↓ 스크롤 유도
│ 급등/급락 6카드 전체     │
│ 주목할만한 종목          │
│ 시장을 움직이는 뉴스     │
```

핵심: 관심종목 영역의 높이를 줄여서(반복 지적했던 "영역 너무 큼" 해결), 급등/급락 카드가 첫 화면에 자연스럽게 걸쳐지도록 한다.

### 내가 배운 것

**스크롤은 "하게 되는 것"이 아니라 "하고 싶게 만드는 것"이다.** 콘텐츠를 보여주는 것과 "보고 싶게 만드는 것"은 다르다. 살짝 걸치는 기법은 유튜브 추천 영상, 인스타그램 피드에서도 쓰이는 패턴이다. 이걸 의식적으로 설계해야 한다.

## Case 35. "관심종목 영역이 너무 크다" — 3번 반복한 이유

### 내가 관찰한 것

관심종목 섹션의 영역 크기에 대해 3번 이상 같은 피드백을 했다:
1. "관심종목이 영역을 너무 크게 잡아먹는 거 아닌가?"
2. "관심종목이 너무 빈 채로 크게 영역을 차지하는데"
3. 스프린트 우선순위에 "관심종목 섹션 크기 줄이기" 직접 포함

### 왜 3번이나 말해야 했는가

처음 말했을 때 반영이 안 됐고, 두 번째 말했을 때도 다른 이슈에 묻혔다. **PO가 같은 피드백을 3번 반복한다는 건, 이 이슈의 우선순위가 팀 내부에서 제대로 인식되지 않고 있다는 의미다.**

### 내 사고 과정

왜 이게 그렇게 중요한가? 단순히 "크기"의 문제가 아니었다:

- **첫 화면 부동산**: 모바일에서 첫 화면은 귀중하다. 관심종목이 50%를 차지하면 나머지 콘텐츠를 볼 수 없다
- **빈 상태 문제**: 종목을 안 넣은 사용자에게는 빈 공간이 50%를 차지하는 것이다
- **다른 이슈의 원인**: "급등/급락이 걸쳐 보여야 한다"는 요구도 관심종목이 크기 때문에 밀려난 거다

**하나의 섹션이 과도한 영역을 차지하면, 다른 콘텐츠의 발견성이 떨어지고, 스크롤 동선이 길어지고, 첫인상이 나빠진다.** 연쇄적 영향이 있는 이슈다.

### 내가 배운 것

**"크기"는 UI의 가장 기본적인 판단이지만, 가장 임팩트가 크다.** 어떤 섹션이 얼마나 큰 영역을 차지하느냐는 곧 "이 서비스에서 무엇이 가장 중요한가?"라는 선언이다. 관심종목이 50%를 차지하면, 사용자는 "이건 관심종목 관리 앱이구나"라고 인식한다. 그게 의도한 포지셔닝인가?

## Case 36. "섹터 로테이션은 별도 메뉴로" — 정보의 깊이에 따른 구조 분리

### 내가 관찰한 것

섹터 로테이션(IT, 금융, 에너지 등 업종별 자금 흐름)이 홈 화면에 풀 섹션으로 들어가 있었다. 데이터가 많고 깊은 콘텐츠다.

### 내 사고 과정

> "섹터 로테이션 같은 건 차라리 별도 메뉴로 빼고, 핫/콜드 일부만 노출해서 별도 메뉴로 유도하면 좋잖아?"

홈 화면의 역할은 **"빠른 파악"**이다. 섹터 로테이션은 깊이 분석하는 콘텐츠라서 홈과 성격이 안 맞는다. 하지만 완전히 숨기면 존재를 모른다 (Case 16에서도 지적).

### 내 판단

정보의 **깊이**에 따라 구조를 분리:

```
홈 (빠른 파악, 1~2초)
  → 핫 섹터 2개, 콜드 섹터 2개만 한 줄 표시
  → "섹터 전체 보기 →" 링크

섹터 페이지 (깊은 분석, 1~2분)
  → 전체 업종 히트맵
  → 기간별 로테이션 추이
  → 업종 내 상위 종목
```

**"훑어보기"와 "깊이 보기"는 다른 화면에서 해야 한다.** 한 화면에 둘 다 넣으면 훑어보기도 불편하고 깊이 보기도 불편하다.

### 내가 배운 것

**콘텐츠의 소비 패턴(글랜스 vs 딥다이브)에 따라 화면을 분리해야 한다.** 홈은 글랜스 전용, 상세는 딥다이브 전용. 이 원칙을 지키면 "홈이 너무 길다" 문제와 "기능을 모른다" 문제를 동시에 해결할 수 있다.

## Case 37. 검색 → 등록 → 가격 표시 — 끊어진 사용자 여정

**핵심 판단 한 줄**: "검색 버튼 없음"이라는 단일 버그가 아니라 4단계 여정 중 3단계가 동시에 끊긴 구조적 문제임을 인식하고, 여정 전체를 한 번에 수정했다.

### 문제 인식

`/`로 종목을 검색하면 결과가 나오는데, 관심종목에 등록하는 버튼이 없었다. 등록 기능 자체도 동작하지 않았다.

> "'/'로 종목검색하면 리스트에 관심종목 등록하는 버튼 추가하고 관심종목 등록 안 되는 거 수정"

### 사용자/서비스 관점

사용자의 기대 여정:
```
1. "카카오 주가 궁금하다" → / 눌러서 검색
2. 검색 결과에서 카카오 발견 → "관심종목에 넣어야지"
3. 등록 버튼 클릭 → 추가됨
4. 홈 화면에서 카카오 실시간 가격 확인
```

관심종목 등록은 이 서비스의 핵심 사용자 여정이다. 이 여정이 끊기면 사용자가 이 앱을 "매수 판단에 쓰는 도구"로 쓸 수 없다. 재방문 이유가 없어진다.

### 사고 과정

처음엔 "등록 버튼이 없다"는 UI 버그로 봤다. 그런데 더 파고드니 단계별로 문제가 쌓여 있었다:

| 단계 | 기대 | 실제 | 원인 |
|------|------|------|------|
| 검색 결과 | 등록 버튼 노출 | 버튼 없음 | UI 누락 |
| 등록 클릭 | 관심종목 추가 | 동작 안 함 | 기능 버그 |
| 등록 후 가격 | 실시간 가격 표시 | 미표시 | polling 대상이 하드코딩 52개 (Case 9) |

2번에서 끊겨도 문제인데, 2번을 고쳐도 3번에서 또 끊긴다. 3번을 고쳐도 4번에서 끊긴다. **입구(검색)는 있는데 출구(실시간 가격)까지 가는 경로 자체가 없는 상태.**

→ 개별 버그 수정이 아니라 **여정 전체 관통 수정**이 필요하다는 판단.

### 내 판단

3가지를 동시에 수정하도록 결정했다:
1. 검색 결과에 관심종목 등록 버튼 추가
2. 등록 로직 수정
3. polling 대상을 하드코딩 52개 → watchlist 기반 동적 목록으로 전환

하나씩 순서대로 수정하면 사용자는 "이제 됩니다 → 아직도 안 됩니다"를 반복 경험한다.

### 결과

3개 동시 수정 후 검색 → 등록 → 가격 표시 여정이 완전히 연결됐다. 이후 "새 종목 등록했는데 가격이 안 나온다"는 피드백이 사라졌다.

### PM으로서 배운 것

**기능 단위로 보면 "검색 OK, 등록 버그, 가격 제한"이다. 여정 단위로 보면 "이 서비스에서 종목 추가가 불가능하다"가 된다.** 각 기능이 개별적으로는 동작해도, 여정으로 연결되지 않으면 사용자에게는 없는 기능이다. QA도, 개발 요청도 기능 단위가 아니라 여정 단위로 해야 이 문제를 잡을 수 있다.

## Case 38. 뉴스 소비 여정 — "하나의 뉴스에서 투자 판단까지"

### 내가 관찰한 것

뉴스를 클릭하면 외부 사이트로 나가버렸다(Case 4). AI 요약은 제목만 반복했다(Case 5). 관련 종목은 BTC 하나만 나왔다(Case 2).

### 내 사고 과정

이 3가지를 각각 수정할 수도 있었다. 하지만 나는 **하나의 통합된 뉴스 소비 여정**을 설계하기로 했다:

> "AI가 그 기사를 요약하고, 연관종목과 함께 관련뉴스도 추려서 보여주는 거 어때?"

이건 3개의 개별 fix가 아니라, 하나의 **뉴스 소비 플로우** 설계다:

```
사용자의 투자 뉴스 소비 여정:

1. 뉴스 제목 발견 (홈 or 뉴스탭)
   → "이거 중요한 건가?"

2. 클릭 → 사이드패널 열림
   → AI 요약 3문장 읽기 (핵심 파악)
   → "내 포트폴리오에 영향 있나?"

3. 관련 종목 확인
   → BTC 뉴스면 IBIT, MSTR, ETH 등 연관 종목 가격 표시
   → "아, ETF도 같이 올랐네"

4. 관련 뉴스 확인
   → 같은 주제의 다른 기사 2~3개
   → "여러 매체에서 다루고 있으니 진짜 중요한 거구나"

5. (선택) 원문 보기
   → 깊이 읽고 싶으면 하단 링크
```

**핵심 설계 원칙: 5단계 중 4단계까지 앱 안에서 완결.** 외부 이동은 마지막 선택지.

### 내 판단

이 여정이 작동하려면 **3가지가 동시에** 동작해야 한다:
1. AI 요약이 실제 기사 본문을 요약해야 함 (데이터 파이프라인)
2. 관련 종목이 투자자 멘탈 모델에 맞아야 함 (relatedAssets 연결)
3. 관련 뉴스가 같은 주제로 묶여야 함 (키워드 매칭 알고리즘)

이 중 하나라도 빠지면 여정이 중간에 끊긴다. **여정은 "가장 약한 고리"의 수준으로 수렴한다.**

### 내가 배운 것

**개별 기능을 나열하면 "AI 요약", "관련 종목", "관련 뉴스" 3개다. 하지만 여정으로 엮으면 "뉴스 하나로 투자 판단하기"라는 하나의 경험이 된다.** 기능을 쌓는 게 아니라 여정을 설계하는 것이 PO의 일이다. 기능은 여정의 재료일 뿐, 완성품이 아니다.

## Case 39. "휴장이면 코인만 나와야지" — 시장 상태에 따른 콘텐츠 적응

### 내가 관찰한 것

토요일 밤에 앱을 열었다. 국장도 미장도 다 닫혀있는데, "주목할 종목" 섹션에 국장/미장 종목이 나오고 있었다.

### 왜 이게 문제라고 느꼈는가

> "주목할 종목은 지금 국장, 미장 다 닫혔는데 코인들이 나오고 있어야 되는 거 아니야?"

**장이 닫혀 있으면 주가가 안 움직인다.** 움직이지 않는 종목을 "주목할만한"이라고 보여주는 건 거짓말이다. 토요일 밤에 "주목할만한" 건 **24시간 움직이는 코인**이다.

### 내 사고 과정: 컨텍스트 적응형 콘텐츠

이 앱은 3개 시장을 다룬다. 각 시장의 운영 시간이 다르다:

```
시장 상태 매트릭스:

              월~금 09:00~15:30   월~금 23:30~06:00   주말/공휴일
국장(KR)      ● 장 중              ○ 장 후              ○ 휴장
미장(US)      ○ 장 후              ● 장 중              ○ 휴장
코인(COIN)    ● 24시간             ● 24시간             ● 24시간
```

**"주목할만한 종목"은 지금 시장이 열려있는 곳의 종목을 우선 보여줘야 한다.** 전부 닫혀있으면 코인만. 미장만 열려있으면 미장 우선. 이런 **컨텍스트 적응**이 없으면, 사용자가 직접 "지금 장이 열려있는 곳"을 판단하고 해당 탭을 찾아가야 한다.

### 내 판단

콘텐츠 노출 우선순위 = 시장 상태 함수:
```javascript
// 의사 코드
if (krOpen) showPriority('KR', 'US', 'COIN')
else if (usOpen) showPriority('US', 'COIN', 'KR_dimmed')
else showPriority('COIN')  // 주말/공휴일
```

이건 "필터" 기능이 아니다. 사용자가 수동으로 탭을 전환하는 게 아니라, **앱이 알아서 지금 의미있는 콘텐츠를 먼저 보여주는 것.** 넷플릭스가 시간대에 따라 추천을 바꾸는 것과 같은 원리다.

### 내가 배운 것

**같은 화면이라도 "언제 보느냐"에 따라 최적의 콘텐츠가 다르다.** 투자 앱은 시장 상태라는 명확한 컨텍스트가 있다. 이걸 활용하지 않으면, 사용자에게 "이 앱은 나를 이해하지 못한다"는 느낌을 준다. 컨텍스트 적응은 개인화의 가장 기본 형태다.

## Case 40. "관심종목의 사용성 유도를 고민해야 해" — 온보딩 메커니즘

### 내가 관찰한 것

관심종목 기능은 이 서비스의 핵심이다. 그런데 신규 사용자의 관심종목은 비어있다(Case 10). 등록 과정도 불편했다(Case 37).

### 내 사고 과정

> "관심종목의 사용성 유도 고민"

이건 단순히 "빈 화면을 채우자"가 아니라, **"사용자가 관심종목을 등록하게 만드는 여정"을 설계**해야 하는 문제다.

사용자가 관심종목을 등록하기까지의 심리 여정:
```
1. 첫 방문 → "이 앱 뭐하는 건데?"
   → 가치 인식이 안 되면 여기서 이탈

2. 둘러보기 → "오 급등/급락 재밌네"
   → 홈의 다른 콘텐츠로 가치를 먼저 증명

3. 특정 종목 관심 → "삼성전자 계속 보고 싶은데"
   → 이 순간에 관심종목 등록 유도가 나와야 함

4. 등록 시도 → "어떻게 넣지?"
   → 검색 → 등록 버튼이 바로 보여야 함 (Case 37)

5. 등록 후 → "오 바로 가격이 나오네"
   → 즉각적 보상이 있어야 재방문 동기
```

### 내 판단

**빈 관심종목 영역 자체가 온보딩 도구가 되어야 한다:**
```
┌─ 관심종목 ──────────────────────┐
│                                 │
│  아직 관심종목이 없어요          │
│                                 │
│  [인기 종목으로 시작하기]        │  ← 원클릭 추가
│                                 │
│  삼성전자  SK하이닉스  테슬라    │  ← 추천 종목 칩
│  비트코인  애플      엔비디아    │
│                                 │
│  또는 / 눌러서 직접 검색하기     │  ← 검색 유도
│                                 │
└─────────────────────────────────┘
```

- **추천 종목 원클릭**: 터치 한 번으로 여러 종목 동시 추가 → 등록 허들 최소화
- **즉각적 보상**: 추가 즉시 가격 + 등락률 표시 → "이게 이 앱의 핵심 가치구나"
- **점진적 유도**: 처음엔 추천으로 시작, 나중에 검색으로 커스텀

### 내가 배운 것

**빈 화면은 위기이자 기회다.** 아무것도 없는 화면을 "빈 상태 메시지"로만 처리하면 위기다. 하지만 온보딩 도구로 설계하면 **사용자를 핵심 기능으로 안내하는 기회**가 된다. 빈 상태 디자인은 서비스의 첫인상이자 사용자 여정의 시작점이다.

## Case 41. 투자 판단 5분 여정 — 기능이 아니라 경험을 설계하다

**핵심 판단 한 줄**: 5개 기능을 나열하는 대신 "매수 판단 5분"이라는 하나의 여정으로 엮고, 모든 기능의 존재 이유를 이 여정 안에서 재평가했다.

### 문제 인식

포지셔닝을 "매수 결정 직전 5분을 함께하는 앱"으로 바꿨다(Case 21). 그런데 기능들은 여전히 개별적으로 존재하고 있었다. 차트, 뉴스, 관련 종목, 이벤트 캘린더 — 다 있는데 연결이 안 됐다. 사용자가 판단을 하려면 여러 기능 사이를 수동으로 오가야 했다.

### 사용자/서비스 관점

투자자가 "매수할까?"를 고민하는 5분의 실제 흐름:

```
0:00  트리거 — "테슬라가 5% 빠졌다는데?"
0:30  현황 파악 — "얼마나 빠졌지? 추세는?" → 차트
1:00  원인 파악 — "왜 빠졌지?" → 뉴스 AI 요약
2:00  맥락 확인 — "테슬라만? 섹터 전체?" → 관련 종목·섹터
3:30  리스크 체크 — "FOMC 며칠 남았지?" → 이벤트 캘린더
4:30  판단 — 앱은 판단을 안 해준다. 재료만 5분 안에 제공.
```

각 단계에서 다른 앱으로 넘어가야 한다면 이 앱의 가치는 없다.

### 사고 과정

기능 리스트 vs 여정 설계의 차이:

| 관점 | 결론 |
|------|------|
| 기능 리스트 | "차트, 뉴스, 관련 종목, 이벤트 캘린더, 섹터 — 다 있다 ✓" |
| 여정 설계 | "이 순서로, 이 타이밍에, 이 맥락에서 자동으로 나와야 한다" |

기능 리스트는 "있느냐 없느냐"를 본다. 여정 설계는 **"연결되어 있느냐"**를 본다.

이 여정 기준으로 모든 기능을 재평가했다:

| 기능 | 여정 내 역할 | 판단 |
|------|-------------|------|
| 차트 + 뉴스 + 관련 종목 | 핵심 (0:30~2:00) | 반드시 연결 |
| 이벤트 캘린더 | 보조 (3:30) | 홈에 요약 노출 필요 |
| 고래 알림 | 간접 기여 | 유지, 우선순위 낮음 |
| DEX 핫 토큰 | 여정과 무관 | **삭제** |

### 내 판단

기능별 우선순위를 여정 순서 기준으로 재배열하고, 여정과 무관한 DEX 핫 토큰을 삭제했다. 이벤트 캘린더는 맨 아래에 있던 것을 홈 상단 롤링(요약)으로 올렸다 — 여정의 3:30 지점에서 자연스럽게 보이도록.

### 결과

뉴스 클릭 → AI 요약 + 관련 종목 + 관련 뉴스가 하나의 사이드패널에 연결됐다. "뉴스 보다가 차트 보고 싶어서 나갔다"는 행동이 줄었다(직접 확인).

### PM으로서 배운 것

**기능은 여정의 재료이지, 완성품이 아니다.** 기능이 다 있어도 여정으로 연결되지 않으면, 사용자는 각 기능 사이를 수동으로 이동해야 한다. 그 이동 비용이 곧 이탈 확률이다. 포지셔닝 → 사용자 여정 → 기능 가치 평가 → 우선순위 결정. 이 순서가 뒤집히면 "기능은 많은데 쓰기 불편한 앱"이 된다.

## Case 42. 뉴스-종목 양방향 연결 — 정보 그래프 설계

### 내가 관찰한 것

뉴스에서 종목으로 이동하는 경로는 만들었다(뉴스 클릭 → 관련 종목 표시). 하지만 반대 방향은 없었다. 종목 차트를 보면서 "이 종목 관련 뉴스가 뭐지?"를 확인할 방법이 없었다.

### 내 사고 과정

투자자의 정보 탐색은 **양방향**이다:
```
뉴스 → 종목: "이 뉴스 때문에 어떤 종목이 움직이지?"
종목 → 뉴스: "이 종목이 왜 이렇게 움직이지?"
```

한 방향만 있으면 탐색이 반쪽이다. 비트코인 뉴스에서 IBIT를 발견하고 클릭 → IBIT 차트 패널이 열림 → 근데 여기서 IBIT 관련 뉴스를 볼 수 없음 → 다시 뉴스 탭으로 돌아가서 검색? **여정이 끊긴다.**

### 내 판단

**"정보 그래프"** 구조:
```
뉴스 A ──→ 종목 X ──→ 뉴스 B
  │           │           │
  ├──→ 종목 Y ├──→ 뉴스 C │
  │           │           │
  └──→ 종목 Z └──→ 뉴스 D └──→ 종목 W
```

어디서 시작하든 관련된 정보로 계속 탐색할 수 있어야 한다. 이건 relatedAssets.js(종목 관계)와 뉴스 키워드 매칭(뉴스-종목 관계)을 **양방향으로** 연결하는 것이다.

구체적으로:
- 뉴스 사이드패널: 관련 종목 표시 (이미 구현)
- 종목 차트 패널: 관련 뉴스 표시 (추가 필요)
- 종목에서 종목: relatedAssets로 연관 종목 (이미 구현)

### 내가 배운 것

**정보를 "나열"하는 서비스와 "연결"하는 서비스는 차원이 다르다.** 나열은 "뉴스 10개, 종목 50개"다. 연결은 "이 뉴스 → 이 종목 → 저 뉴스 → 저 종목"으로 사용자가 자유롭게 탐색하는 그래프다. 투자 대시보드의 진짜 가치는 개별 정보가 아니라 **정보 간 연결**에 있다.

## Case 43. "시장을 움직이는 뉴스" — 콘텐츠 네이밍의 힘

### 내가 관찰한 것

뉴스 섹션의 이름을 여러 번 바꿨다. "투자 시그널 뉴스" → 여러 후보 논의 → "시장을 움직이는 뉴스"로 결정.

### 내 사고 과정

> "변경된 콘텐츠 (시장을 움직이는 뉴스 등)는 좋은 것 같아"

이 긍정 피드백은 **이름이 콘텐츠의 기대치를 정확히 설정**했기 때문이다. 이름별 사용자 기대치 비교:

```
"투자 시그널 뉴스"
  → "시그널? AI가 매수/매도 신호를 주나?" → 과한 기대 → 실망

"오늘의 핵심뉴스"
  → "에디터가 고른 건가?" → 큐레이션 기대

"시장을 움직이는 뉴스"
  → "이 뉴스 때문에 시장이 움직인 거구나" → 인과관계 기대 → 정확히 우리가 하는 것
```

**"시장을 움직이는"이라는 수식어가 콘텐츠의 선별 기준을 사용자에게 명확히 전달한다.** 스포츠 뉴스, 연예 뉴스가 여기 나오면 이름과 안 맞으니까 자연스럽게 필터링 기준이 된다.

### 내가 배운 것

**콘텐츠 네이밍은 UI 텍스트가 아니라 "약속"이다.** "시장을 움직이는 뉴스"라고 이름 붙이는 순간, 이 섹션에 나오는 모든 뉴스는 "시장을 실제로 움직인" 것이어야 한다. 이름이 콘텐츠의 품질 기준이 되고, 알고리즘의 방향이 되고, 사용자의 신뢰가 된다.

## Case 44. 10개 이슈 일괄 피드백 — "서비스를 사용자처럼 써보다"

### 내가 관찰한 것

어느 날 앱을 쭉 사용하면서 10개 이슈를 한 번에 발견했다:

1. "이유분석중" 영구 로딩
2. AI 요약이 제목 반복
3. 핵심뉴스 클릭 시 인터랙션 불일치
4. 이벤트 캘린더 발견 불가
5. 특정 종목 차트 미표시
6. 업비트 대량체결 누락 + 고래 기준
7. "뉴스나왔지만 미반응" 섹션 무용
8. 코인 로고 이미지 깨짐
9. 관심종목 사용성 유도 부재
10. 검색 → 관심종목 등록 불가

### 내 사고 과정

이 10개를 한 번에 발견한 건 우연이 아니다. **"사용자처럼 써봤기 때문"**이다.

PO가 자기 서비스를 "테스트"하는 것과 "사용"하는 것은 다르다:
- **테스트**: "이 버튼 누르면 동작하나?" → 기능 검증
- **사용**: "토요일 밤에 내일 뭐 사지?" → 실제 상황에서의 여정

나는 주말 밤에 "내일 장 열리면 뭘 볼까?"라는 실제 투자 상황으로 앱을 열었다. 그랬더니:
- 휴장인데 국장 종목이 나옴 (2, 5번 → 컨텍스트 부적응)
- 뉴스를 봐도 판단 재료가 안 됨 (1, 2, 3번 → 뉴스 여정 미완성)
- 관심종목 자체가 비어있음 (9, 10번 → 온보딩 여정 부재)
- 부가 정보가 신뢰 안 됨 (5, 6, 8번 → 데이터 품질)
- 불필요한 노이즈 (7번 → 기능 가치 부재)

### 내 판단

**정기적으로 "사용자 모드"로 앱을 써야 한다.** 개발 중에는 코드만 보지, 사용자 여정을 체험하지 않는다. 일주일에 한 번은 "지금 시장 보자"라는 실제 상황으로 앱을 열어보는 것이 필요하다.

### 내가 배운 것

**최고의 사용자 리서치는 PO가 직접 사용자가 되는 것이다.** 설문, 인터뷰, 데이터 분석도 중요하지만, 실제 상황에서 자기 서비스를 써보면 **설문에서 안 잡히는 여정 단절, 컨텍스트 불일치, 감정적 불만**을 직접 느낄 수 있다. "나도 불편하면 사용자는 더 불편하다"가 기본 가설이다.

## Case 45. 정보 계층 3단계 — "글랜스 → 요약 → 상세"

### 내 사고 과정

여러 케이스를 겪으면서, 정보를 보여주는 데 **3개의 깊이 수준**이 필요하다는 패턴을 발견했다:

```
Level 1: 글랜스 (Glance) — 0.5초
  홈 화면에서 한 눈에 파악
  "시장 괜찮나?" "내 종목 괜찮나?"
  → 숫자, 색깔, 방향(↑↓)만으로 전달

Level 2: 요약 (Summary) — 5~10초
  사이드패널에서 빠르게 읽기
  "왜 올랐지?" "이 뉴스 핵심이 뭐지?"
  → AI 요약, 관련 종목, 1문장 이유

Level 3: 상세 (Detail) — 1분 이상
  별도 페이지나 외부 링크
  "전체 기사 읽기" "섹터 전체 히트맵"
  → 원문 링크, 전체 차트, 종합 분석
```

### 이걸 깨달은 계기

- **섹터 로테이션**(Case 36): 홈에 전부 보여주려 함 → Level 1과 Level 3을 섞으면 둘 다 망함
- **이벤트 캘린더**(Case 27): 맨 밑에 Level 3를 넣음 → Level 1(롤링 요약)이 필요
- **뉴스**(Case 38): 외부 이동(Level 3만) → Level 2(인앱 요약) 추가 필요
- **관심종목**: 가격/등락률(Level 1) → 차트 패널(Level 2) → 상세 페이지(Level 3)

### 내 판단

**모든 콘텐츠를 이 3단계로 설계:**

| 콘텐츠 | Level 1 (홈) | Level 2 (패널) | Level 3 (상세) |
|--------|-------------|---------------|---------------|
| 종목 | 가격 + 등락% | 차트 + 관련종목/뉴스 | 종합 분석 (미구현) |
| 뉴스 | 제목 1줄 | AI 요약 + 관련종목 | 원문 기사 |
| 섹터 | 핫/콜드 2개 | (미구현) | 전체 히트맵 |
| 이벤트 | "FOMC D-3" 롤링 | 모달 캘린더 | (미구현) |
| 지수 | 숫자 + 등락% | (불필요) | (불필요) |

### 내가 배운 것

**"얼마나 보여주느냐"가 아니라 "어떤 깊이로 보여주느냐"가 정보 설계의 본질이다.** Level 1에 Level 3 콘텐츠를 넣으면 "홈이 너무 길다." Level 1만 있으면 "정보가 부족하다." 3단계를 명확히 나누고, 사용자가 원할 때 깊이 들어갈 수 있게 경로를 설계하는 것이 정보 아키텍처다.

## Case 46. 클래러티 법안 뉴스에 관련종목이 안 나왔다 — 영어 키워드 갭 (2026-03-29)

### 내가 관찰한 것

"Clarity Act", "FIT21", "stablecoin bill" 같은 미국 암호화폐 규제 뉴스를 클릭했는데 관련 종목이 하나도 안 나왔다. 코인베이스(COIN), 서클, MicroStrategy 같은 회사들이 당연히 나와야 하는데 빈 패널이었다.

### 왜 이게 문제라고 느꼈는가

> "이걸 왜 니네는 발견 못해? 그리고 이러한 이슈가 지금 약 10번은 더 넘게 지속되고 있음."

단순한 버그가 아니었다. 뉴스-종목 연결 알고리즘 자체의 구조적 결함이었다. 키워드 사전(`newsTopicMap.js`)이 한국어만 있었다. 영어 뉴스 제목을 처리하면 아무것도 감지 안 됐다.

### 사고 과정

문제 구조:
```
뉴스 제목: "Clarity Act passes committee vote"
  → detectNewsSectors("clarity act passes committee vote")
  → 한국어 키워드만 있음 → 매칭 0개
  → 관련 섹터 없음 → 관련 종목 없음 → 빈 패널
```

추가로 발견한 문제: COIN(코인베이스), MSTR(MicroStrategy)이 `relatedAssets.js`에 최상위 항목으로 없었다. 다른 종목의 `related` 배열 안에만 있어서 직접 검색이 안 됐다.

### 내 판단

두 가지 동시 수정:
1. `newsTopicMap.js` 암호화폐 키워드에 영어 + 규제 법안 용어 추가 (`clarity act`, `fit21`, `stablecoin bill` 등)
2. `relatedAssets.js`에 COIN, MSTR을 최상위 항목으로 추가 (섹터: '암호화폐기업')

### 내가 배운 것

**서비스가 한국어 사용자를 타겟으로 해도, 데이터는 글로벌이다.** 미국 규제 뉴스는 영어 제목으로 들어온다. 키워드 사전을 한국어만 관리하는 건 "국내 뉴스만 제대로 연결된다"는 뜻이다. 데이터 파이프라인은 항상 실제 입력 데이터 형태를 기준으로 설계해야 한다.

## Case 47. "비표준"이라는 말로 의도된 설계가 지워졌다 — 문서화의 본질 (2026-03-29)

### 내가 관찰한 것

Vercel 배포가 main 머지마다 자동 실행되고 있었다. 예전에 수동 배포 게이트(GitHub Actions `workflow_dispatch`)를 만들었는데 어느 순간 사라져 있었다. PR #192 히스토리를 파보니, Claude가 "비표준 GitHub Actions 워크플로우 제거"라고 직접 지웠고, Copilot이 "표준화 작업이네요"라며 승인했다.

### 왜 이게 문제라고 느꼈는가

여러 프로젝트(chimp-pick, signalplay, costock, aptner 등)가 Vercel Pro 6,000 builds/월을 나눠 쓴다. PR 머지마다 배포가 나가면 금방 한도가 찬다. 수동 배포 게이트는 이걸 막기 위한 의도된 설계였다.

### 사고 과정

왜 지워졌나:
- 코드를 보면 "비표준 방식"처럼 보임 (Vercel 기본값이 Git 통합 자동 배포이니까)
- "왜 이렇게 만들었는지"가 어디에도 없었음
- AI는 비표준 패턴을 보면 정리하려는 경향이 있음
- Copilot도 "표준화"라며 승인 → 아무도 막지 않음

### 내 판단

기술 설정만으로는 재발 방지가 안 된다. "왜"를 5곳에 동시에 박아야 한다:
1. 코드 자체 (워크플로우 파일 상단 주석)
2. ADR (decisions.md — ADR-013)
3. 마스터 문서 (CLAUDE.md)
4. 메모리 (AI가 다음 대화에서도 기억)
5. 커밋 메시지 (히스토리에 이유 남김)

### 내가 배운 것

**"비표준"은 나쁜 게 아니다. 의도된 비표준은 설계다.** Vercel 기본값(main 머지 = 자동 배포)은 개인 프로젝트 기준이다. 다중 프로젝트 운영, 배포 횟수 최적화가 필요한 환경에서는 수동 게이트가 더 성숙한 방식이다. 플랫폼의 기본값이 우리 맥락에 항상 맞는 건 아니다.

## Case 48. 주말 휴장인데 국장/미장을 같이 보여줄까? — 실시간 vs 마감 데이터 공존 (2026-03-29)

### 내가 관찰한 것

주말에 "주목할만한 종목" 섹션을 열면 코인만 나온다. Case 39에서 "휴장이면 코인만 나와야지"라고 결론 냈었다. 근데 다시 생각해보니 "금요일 마감이 어땠는지, 다음 주 뭘 봐야 하는지"도 같이 뜨면 좋겠다는 고민이 생겼다.

### 왜 다시 고민하게 됐는가

주말에 앱을 여는 사람의 니즈가 두 가지다:
1. **지금 움직이는 것** (코인 실시간)
2. **지난 거래일 복기** (금요일 국장/미장 마감 흐름, 다음 주 주목 종목)

Case 39의 결론은 1번만 해결했다. 2번은 여전히 빠져 있다.

### 사고 과정

두 니즈는 충돌하지 않는다. 구분만 명확하면 같이 보여줄 수 있다:

```
[지금 주목] LIVE 배지 → 코인 실시간 종목
[마지막 거래일] 마감 배지 → 금요일 국장/미장 상위 종목
```

많은 투자 앱이 이미 이렇게 한다. 주말에도 금요일 종가 기준 "이번 주 주목 종목"을 보여주고 `마감 기준`으로 라벨링.

단, 구현 전 확인 필요: 현재 "주목할만한" 알고리즘이 실시간 수급 이상 감지 기반이라면, 거래 없는 마감 데이터로 동일하게 의미 있는 결과가 나오는지 검증 필요.

### 내 판단

Case 39를 뒤집는 게 아니라 **확장**이다. "실시간 우선 + 마감 데이터 보조"로 레이어를 나눈다. P2로 백로그 등록 예정.

### 내가 배운 것

**한 번 내린 결론도 사용 맥락이 바뀌면 재검토해야 한다.** "휴장이면 코인만"은 "지금 실시간으로 의미 없는 데이터를 보여주지 말자"는 원칙이었다. 근데 마감 데이터는 의미 없는 게 아니라 다른 종류의 의미가 있다. 원칙의 의도를 지키면서 활용 범위를 넓히는 것이 진화다.

## Case 33 — PR마다 배포하면 한도가 초과된다 (2026-03-25)

**관찰한 것:** Vercel 무료 플랜 100회/일 한도 초과로 배포가 막혔다.

**왜 이게 문제라고 느꼈는가:** "메인에 100회나 배포하지 않았을 것 같은데"라고 말했다. 맞다. main 머지는 몇 번 안 됐다. 그런데 PR push마다 Preview 배포가 자동 생성되고, 거기에 CLI 수동 배포까지 겹쳤다.

**내 판단:** Vercel Preview 배포 API로 비활성화. 배포는 main 머지 시 GitHub 연동 자동 배포만. CLI 수동 호출 금지.

**배운 것:** 비개발자 입장에서 "배포 횟수"의 개념이 다르다. PR push = 배포라는 걸 모를 수 있다. 팀이 Vercel 무료 플랜 구조를 명확히 알고 있어야 한다.

## Case 34 — API 있어도 연결 안 하면 의미 없다 (2026-03-25)

**관찰한 것:** `/api/us-price` Yahoo v8 실시간 프록시가 존재했지만 미장 changePct는 계속 0이었다.

**왜 이게 문제라고 느꼈는가:** 코드는 완성됐는데 사용자에게 전달이 안 됐다. `fetchUsStocksBatch`가 여전히 Stooq 클라이언트 직접 호출을 1순위로 쓰고 있었다.

**내 판단:** 새 API 엔드포인트 만들 때 프론트 연결까지 한 세트. QA 체크리스트에 "API 응답 → UI 반영" 확인 항목 필수.

**배운 것:** 인프라와 UI는 별개 작업처럼 보이지만, 연결이 빠지면 둘 다 의미 없다. 작업 완료 기준은 "사용자 화면에 보이는 것"이어야 한다.

## Case 35 — signalplay 구조를 차용한다 (2026-03-25)

**관찰한 것:** signalplay 프로젝트에 Claude Code가 스스로 생각하고 실행하는 자율주행 구조가 있었다. autopilot-loop, quality ratchet, request-to-ceo, 6명 에이전트 팀.

**왜 이게 필요하다고 느꼈는가:** 매번 "이거 해줘"라고 지시하는 구조보다, 팀이 스스로 backlog를 보고 다음 작업을 판단하는 구조가 더 지속 가능하다.

**내 판단:** market-dashboard-v2에 동일 구조 이식. autopilot-loop (무한 루프), quality-baseline.md (래칫), request-to-ceo (CEO 인터페이스), 나머지 스킬 체인 추가.

**배운 것:** "자율"의 정의 = CEO 개입 없이도 품질이 지속적으로 향상되는 상태. 이게 되려면 팀이 읽을 수 있는 문서(backlog, decisions, quality-baseline)와, 판단 기준(HARD RULES), 그리고 루프가 필요하다.
