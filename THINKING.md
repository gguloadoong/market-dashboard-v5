# Market Radar — 나의 프로덕트 사고 기록

> 투자 시세 모니터링 대시보드를 기획·운영하며 내가 어떤 문제를 발견하고,
> 어떤 질문을 던지고, 어떻게 판단했는지를 기록한다.
>
> **역할**: Product Owner
> **기간**: 2026.03 ~
> **제품**: Market Radar — 국내주식·미국주식·암호화폐 통합 실시간 대시보드

## Case 1. "지수를 못 받아오는 건 니가 문제 아니야?"

### 내가 관찰한 것

개발팀에 "시장 지수가 누락된다"고 보고했더니, 돌아온 답은 "Vercel 환경변수가 설정되었는지 확인해달라"였다. 이미 설정되어 있다고 여러 번 말했는데도 같은 질문이 반복됐다.

### 왜 이게 문제라고 느꼈는가

개발팀이 문제의 원인을 외부(환경 설정)에서 찾고 있었다. 하지만 나는 실제로 앱을 쓰면서 "어떤 지수는 나오고 어떤 건 안 나온다"는 패턴을 봤다. 설정 문제라면 전부 안 나와야 정상이다. **일부만 실패하는 건 코드 로직 문제다.**

### 내 판단

> "버셀에 키가 있대도..? 왜 자꾸 물어보는거야"
> "지수를 못 받아오는 니가 문제 아니야? 개발팀 QA 이거 원인파악해 그리고 CTO 영입해 그냥"

단순한 버그 리포트가 아니라, **시스템 전체를 감사해야 한다**고 판단했다. 이유:
- 같은 종류의 데이터 이슈가 반복 발생하고 있었다 (등락률 0%, 지수 누락, sparkline 빈칸)
- 개별 fix로는 다른 곳에서 또 터질 것 같았다
- "CTO급 전수 감사"를 요청한 건, 증상이 아니라 구조를 봐달라는 의미였다

### 결과

감사 결과 실제로 코드 문제였다:
- NDX 지수가 `^IXIC`(나스닥 종합, 18,000)을 호출하면서 UI에는 "NDX"(나스닥-100, 21,000)로 표시
- 등락률 비교 함수가 4개 파일에 중복되어 있고, 모두 같은 방식으로 잘못됨
- SSRF 보안 취약점 3건 발견

### 내가 배운 것

**개발팀이 "환경 설정 확인해달라"고 할 때, 그건 코드 안에서 원인을 못 찾았다는 신호일 수 있다.** PO로서 "그건 이미 했다"고 명확히 끊고, 조사 방향을 전환시키는 것도 역할의 일부다. 그리고 반복되는 문제에는 "한 번 더 고쳐줘"가 아니라 "전체를 봐"라고 에스컬레이션하는 판단이 필요하다.

## Case 2. "비트 뉴스면 비트만 보여주면 안 되지"

### 내가 관찰한 것

비트코인 관련 뉴스를 클릭했을 때, 관련 종목에 BTC 하나만 표시됐다. 재건축 수주 뉴스를 클릭했을 때 건설사가 하나도 안 나왔다.

### 왜 이게 문제라고 느꼈는가

투자자가 "비트코인 급등" 뉴스를 보면 머릿속에 떠오르는 건 BTC만이 아니다:

> "비트 관련 뉴스면 비트 말고 메이저코인 몇 개나 스트래티지(MSTR), BTC ETF 같은 건 보여야지? 코인베이스나..? 재건축 수주 뉴스에 건설사 하나도 없는 것도 이상하고. 무슨 말인지 알지?"

나는 투자자로서의 사고 흐름을 알고 있다. BTC 뉴스를 보면 자연스럽게 "IBIT는 어때?", "MSTR은?", "ETH도 같이 올랐나?"를 확인한다. 이 연쇄적 탐색이 제품 안에서 끊기고 있었다.

### 내 판단

문제는 "검색 알고리즘이 약하다"가 아니라, **투자자의 멘탈 모델과 제품의 데이터 연결 구조가 불일치**한 것이었다.

확인해보니 코드에 `relatedAssets.js`라는 948줄짜리 연관 자산 매핑 파일이 이미 있었다. BTC → IBIT, MSTR, COIN, ETH 등 15개 관계가 정의되어 있었다. **하지만 뉴스 매칭에서는 이걸 전혀 안 쓰고 있었다.** 차트 상세 패널에서만 사용 중이었다.

### 해결 방향

"새로운 관계 데이터를 만들어라"가 아니라, **이미 있는 데이터를 올바른 곳에 연결하라**.
- 뉴스 → 키워드 직접 매칭(기존) → 매칭된 종목의 연관자산 자동 확장(추가)
- BTC 매칭 → IBIT, MSTR, COIN, ETH 등 최대 8개까지 표시

### 내가 배운 것

**기능이 없는 게 아니라 연결이 안 되어 있을 때가 더 아깝다.** 관계 데이터 948줄을 이미 공들여 만들어놓고 한 곳에서만 쓰고 있었다. PO가 데이터 흐름 전체를 보지 않으면 이런 "자원 낭비"를 놓치게 된다.

## Case 3. "휴장이면 왜 종목이 아예 안 보여?"

### 내가 관찰한 것

국내 장이 닫히면 "국내 급등" 섹션에 "🌙 장마감"이라는 오버레이만 뜨고, 종목 리스트가 완전히 사라졌다.

### 왜 이게 문제라고 느꼈는가

> "국장/미장 급등급락 '휴장' 표시까진 좋은데 왜 종목들이 아예 안 보여? 딤 처리되고 장마감 기준으로는 있어야 되는 거 아니야?"

나는 **장이 닫혀도 "어제 뭐가 올랐지?"를 확인하고 싶다.** 다음 날 장 시작 전에 어제의 급등/급락을 복기하는 건 트레이더의 기본 루틴이다. 그런데 이 데이터가 있는데도 "휴장"이라는 이유로 통째로 숨기고 있었다.

### 내 판단

- "데이터가 없어서 못 보여주는 것"과 "데이터가 있는데 숨기는 것"은 완전히 다르다
- 휴장 상태 표시는 필요하지만, 그 방법이 "전체 숨김"이어서는 안 된다
- **딤 처리(흐리게) + 작은 라벨**이면 "이건 실시간이 아니다"를 전달하면서 데이터는 유지할 수 있다

### 해결 방향

오버레이(종목 전체 대체) → 딤 처리(opacity 60%) + 인라인 라벨("🌙 장마감 · 전일 종가 기준")
- 4개 섹션(국내 급등/급락, 미장 급등/급락) 모두 동일하게 적용
- 코인은 24시간이므로 변경 없음

### 내가 배운 것

**"없음"을 표현하는 방식이 UX의 수준을 결정한다.** 빈 화면, 로딩 스피너, 에러 메시지, 딤 처리 — 각각이 사용자에게 전달하는 의미가 다르다. "데이터가 있는데 실시간이 아닌 상태"에는 "흐리게 보여주기"가 맞다.

## Case 4. "뉴스 눌렀는데 왜 밖으로 나가?"

### 내가 관찰한 것

뉴스 항목을 클릭하면 외부 기사 링크로 바로 이동했다. 사용자가 앱을 떠나버렸다.

### 왜 이게 문제라고 느꼈는가

> "뉴스는 간략하게 요약, 그리고 관련 종목들을 노출시키는 방향으로. 종목상세처럼 스윽 나오게"

투자자가 뉴스를 클릭하는 이유는 "기사를 정독"하기 위해서가 아니다. **"이 뉴스가 내 포트폴리오에 영향을 주는가?"**를 빠르게 판단하기 위해서다. 외부 이동은 이 판단 흐름을 끊는다.

### 내 판단

뉴스 클릭 시 인앱 사이드 패널을 열어서:
1. AI가 기사를 3문장으로 요약
2. 관련 종목 실시간 가격 표시
3. 같은 주제의 관련 뉴스 추가 노출
4. 원문이 필요하면 하단 "원문 보기" 링크

기존 차트 사이드 패널과 동일한 슬라이드 인터랙션을 쓰면 사용자가 이미 익숙한 패턴이다.

### 내가 배운 것

**사용자가 앱을 떠나는 순간 = 서비스가 가치를 제공할 기회를 잃는 순간.** 외부 링크가 아니라 인앱 경험으로 전환하는 건 단순 UI 변경이 아니라 "사용자의 판단 흐름을 우리 안에서 완결시키겠다"는 전략적 결정이다.

## Case 5. "제목을 따라가버리네?"

### 내가 관찰한 것

AI 뉴스 요약 기능을 달았는데, 결과물이 기사 제목을 반복하는 수준이었다.

### 왜 이게 문제라고 느꼈는가

> "뉴스 요약이 제대로 동작하지 않는 것 같아. 기사 글 전문이 몇 줄로 깔끔하게 요약되어야 하는데, 그냥 제목을 따라가버리네?"

제목만 반복하면 요약이 아니다. "AI 요약"이라는 라벨을 달아놓고 제목을 앵무새처럼 반복하면 **사용자가 AI 기능 전체를 신뢰하지 않게 된다.**

### 내 판단

원인을 추적해보니, RSS에서 받은 `description` 필드(보통 1~2문장)만 AI에 넘기고 있었다. **기사 본문을 크롤링하지 않고 있었던 것.** 입력이 제목+1문장이면 출력도 그 수준을 벗어날 수 없다.

해결 방향:
1. 기사 URL로 HTML 크롤 → `<article>` 태그 또는 `<p>` 태그에서 본문 추출
2. 크롤 실패 시 제목+RSS description 조합으로 fallback
3. 프롬프트도 분리: 본문이 있으면 "핵심 3문장 요약", 없으면 "이 정보로 투자 시사점 분석"

### 내가 배운 것

**AI 기능의 품질은 모델이 아니라 입력 데이터가 결정한다.** GPT-4든 Gemini든, 제목 1줄 넣으면 제목 수준의 답이 나온다. "AI 요약이 별로야"라는 피드백의 80%는 프롬프트나 모델이 아니라 **데이터 파이프라인** 문제다.

## Case 6. "무료 모델 없어?"

### 내가 관찰한 것

AI 요약 기능을 위해 API 키가 필요했다. 유료 모델 목록을 보여줬더니 바로 질문이 나왔다.

### 내 사고 과정

> "무료 모델 없어?"

단순히 돈을 아끼려는 게 아니었다. 이 서비스는 **개인 프로젝트**다. 유료 API에 의존하면:
- 매월 고정 비용 발생 → 서비스 유지 부담
- 사용량 증가 시 비용 예측 불가
- 키 관리 포인트 증가

Gemini Flash의 무료 티어(분당 15회, 일 1,500회)를 발견하고, 이 서비스의 뉴스 요약 호출량(하루 50~100회 추정)과 대조해서 무료로 충분하다고 판단했다.

### 내가 배운 것

**비용 구조를 초기에 결정하는 것은 기술 선택이 아니라 비즈니스 판단이다.** "일단 유료로 가고 나중에 최적화"는 개인 프로젝트에서 서비스 종료 사유가 된다. 무료 티어의 한계를 정확히 계산하고, 그 안에서 동작하도록 설계하는 것이 지속 가능한 서비스의 조건이다.

## Case 7. "Gemini 키 rate limit인데 요약은 왜 잘 되지?"

### 내가 관찰한 것

GitHub Actions에서 "Gemini API rate limit exceeded"라는 에러가 떴다. 그런데 프로덕션 앱에서 뉴스 요약 기능(같은 Gemini API 사용)은 정상 동작하고 있었다.

### 왜 이게 이상하다고 느꼈는가

> "Gemini API 키 rate limit 걸렸다는데 뉴스 눌러보면 요약 잘 되고 있는데, API 키에 무슨 일이 있는지 확인해줄래?"

같은 API 키가 rate limit에 걸렸는데 다른 곳에서는 잘 된다? 논리적으로 둘 중 하나다:
1. 키가 다르다
2. 같은 키인데 하나는 캐시로 버티고 있다

### 결과

조사 결과 **같은 API 키를 두 곳에서 공유**하고 있었다:
- 프로덕션 뉴스 요약 (Vercel Edge Function)
- PR 리뷰 봇 Qodo (GitHub Actions)

짧은 시간에 PR 3개를 연속 생성하면서 Qodo가 API를 과도하게 호출 → rate limit 도달. 프로덕션 뉴스는 **1시간 CDN 캐시**가 있어서 rate limit 상황에서도 캐시된 응답을 반환하고 있었다.

### 내 판단

Qodo를 제거하기로 결정했다. 이유:
- 프로덕션 서비스와 CI 도구가 같은 API 키를 공유하는 구조 자체가 위험
- Qodo의 가치 대비 프로덕션 뉴스 요약 기능이 더 중요
- 리뷰어를 4명에서 3명(Gemini, CodeRabbit, Copilot)으로 줄여도 품질 유지 가능

### 내가 배운 것

**"에러가 난 곳"과 "에러의 원인"이 다를 수 있다.** rate limit 에러는 GitHub Actions에서 났지만, 진짜 위험은 프로덕션 뉴스 요약이 죽는 것이었다. "지금 잘 되고 있으니 괜찮다"가 아니라, "캐시가 만료되면 같이 죽겠구나"를 먼저 봐야 한다.

## Case 8. "카톡으로 계속 토큰 발급 알림이 와"

### 내가 관찰한 것

한국투자증권 API를 쓰는데, 내 핸드폰에 카카오톡 알림이 끊임없이 왔다. "API 접근 토큰이 발급되었습니다."

### 왜 이게 문제라고 느꼈는가

> "지금 한국투자증권 API 호출할 때마다 토큰을 매번 새로 발급받아서 내 핸드폰에 카톡 알림 테러가 오고 있어."

서비스의 기술적 문제가 **내 일상생활을 방해**하고 있었다. 30초마다 가격을 polling하니까 30초마다 토큰 발급 → 30초마다 카톡 알림. 토큰 유효기간이 24시간인데 매번 새로 발급받을 이유가 없다.

### 내 판단

> "토큰 유효기간(24시간) 동안은 재사용할 수 있도록 토큰 캐싱 로직을 추가해 줘. 만료됐을 때만 재발급받도록 수정해 줘."

문제 정의부터 해결 방향까지 직접 제시했다. 이건 개발팀에게 "알아서 해줘"가 아니라, 어떻게 고쳐야 하는지 구체적으로 말해야 빠르게 해결된다고 판단했다.

### 후속

수정 후에도 15분 뒤 다시 알림이 왔다. 캐싱이 Edge Function 특성상 인스턴스 간 공유가 안 되는 문제였다. **1차 수정이 됐다고 끝이 아니라, 실제로 알림이 멈췄는지 검증해야 한다.** 이후 Serverless 런타임으로 전환해서 해결.

### 내가 배운 것

**사용자 경험은 화면 안에만 있지 않다.** API 호출 한 번이 사용자의 카톡 알림으로 이어질 수 있다. 기술적 의사결정의 영향 범위를 "우리 앱 안"으로 한정하면 놓치는 것들이 있다.

---

## Case 9. "한투 API가 있는데 왜 전종목을 못 보여줘?"

### 내가 관찰한 것

한국투자증권 API 키를 세팅했고, 일부 종목은 잘 나온다. 그런데 검색해서 추가한 종목은 가격이 안 나온다. 코스피·코스닥 전종목 + ETF를 커버하지 못하고 있었다.

### 내 사고 과정

> "한국투자증권 API가 있는데도 코스피, 코스닥, 국장에 있는 ETF의 전종목과 가격을 반영하지 못하는 이유 분석해줘"

확인해보니 **코드에 종목 52개가 하드코딩**되어 있었다. 새 종목을 검색해서 watchlist에 추가해도, polling 대상은 항상 그 52개뿐이었다. API는 있는데 코드가 활용하지 않고 있었다.

> "어떻게 해야 전종목을 (미장 포함) 노출할 수 있을까? 한투 아니어도 좋으니 방법을 찾아줘. 유료여도 좋으니"

한투 API에 고정되지 않았다. **목표는 "전종목 커버"이고, 수단은 유연하게** 선택할 수 있다.

### 내 판단

미장 데이터의 경우 "15분 지연"이 아닌 실시간 소스를 원했다:

> "미국 15분 지연 아닌 거 찾아줘"

결과적으로 Yahoo v8(실시간) → Stooq(fallback) → 개별 chart(최종 fallback) 3단 구조로 결정. 국장은 watchlist 기반 동적 polling으로 변경해서, 검색 → 추가 → 가격 표시가 자연스럽게 이어지도록 했다.

### 내가 배운 것

**"API가 있다"와 "API를 활용하고 있다"는 다른 말이다.** 기능이 있는데 안 쓰이고 있으면 그건 기능이 없는 것과 같다. PO는 "이 기능 넣었습니다"를 들었을 때 "근데 실제로 동작해?"를 반드시 확인해야 한다.

---

## Case 10. "처음 들어오면 아무것도 없는 걸 보게 되잖아?"

### 내가 관찰한 것

관심종목 섹션이 화면의 큰 영역을 차지하는데, 신규 사용자는 아무 종목도 등록하지 않은 상태다. 빈 영역만 크게 보인다.

### 왜 이게 문제라고 느꼈는가

> "관심종목이 너무 빈 채로 크게 영역을 차지하는데, 이 서비스에 처음 들어오면 아무것도 없는 걸 보게 되잖아?"

이건 **첫인상 문제**다. 투자 앱에 처음 들어왔는데 빈 화면이면 "이 앱 콘텐츠가 없나?"라고 느낀다. Cold Start 상태에서의 UX를 설계하지 않은 것이다.

### 내 판단

- 관심종목 영역 크기를 줄여야 한다 (3번 반복 지적한 이슈)
- 빈 상태에서는 "인기 종목 추천" 같은 온보딩 콘텐츠를 보여줘야 한다
- 또는 기본 종목을 미리 넣어두고 사용자가 커스텀하게 해야 한다

### 내가 배운 것

**서비스의 첫 화면은 "가장 많은 데이터가 있을 때"가 아니라 "아무 데이터도 없을 때"를 기준으로 설계해야 한다.** 기존 사용자 기준으로만 보면 Cold Start 문제를 놓치고, 신규 사용자 이탈률이 높아진다.

---

## Case 11. "나라면 핫한 거 먼저 궁금할 것 같고"

### 내가 관찰한 것

홈 화면에 6개 섹션이 동일한 크기와 무게로 나열되어 있었다. 어떤 섹션이 더 중요한지 시각적 위계가 없었다.

### 내 사고 과정

> "이 서비스 처음 오면 나라면 주목할 만한 (결국 핫한 거) 궁금할 것 같고, 관심종목과 투자 시그널은 훑어볼 것 같아. 아무래도 주목할 만한 - 관심종목 이 2가지에 포커스가 갈 것 같고"

나 자신을 사용자로 놓고 **행동 시퀀스를 시뮬레이션**했다:
1. 앱 열면 → "오늘 뭐가 핫하지?" (주목할 만한)
2. 그다음 → "내 종목은?" (관심종목)
3. 시간 되면 → "무슨 뉴스?" (투자 시그널)

이 순서가 곧 레이아웃 우선순위여야 한다.

### 내 판단

> "급등/급락 6카드 섹션이 살짝 첫 접속 화면에서 걸쳐 보여야 좋은 듯?"

Above-the-fold 전략: 첫 화면에서 핫 종목 카드가 "살짝 걸쳐" 보이면 스크롤 동기가 생긴다. 완전히 보이거나 완전히 안 보이는 것보다, **일부만 보이는 게** 탐색 욕구를 자극한다.

> "투자 시그널 뉴스는 다른 타이틀명과 뉴스 선별도 좀 더 알고리즘 강화하고, 노출 방식도 단순 리스트 말고 뭔가 비주얼화하면 좋을 것 같아. (종목이랑 연계하거나?)"

뉴스를 단순 리스트가 아닌, 종목과 연결된 비주얼 카드로 바꾸자는 제안.

### 내가 배운 것

**레이아웃은 "예쁘게 배치"가 아니라 "사용자의 행동 순서를 반영"하는 것이다.** 내가 이 앱을 열 때 뭘 먼저 보는지를 관찰하면, 어떤 섹션이 위에 와야 하는지가 자연스럽게 결정된다.

---

## Case 12. "뉴스나왔지만 미반응? 용도를 모르겠음"

### 내가 관찰한 것

홈 화면에 "뉴스가 나왔지만 주가 미반응" 섹션이 있었다.

### 왜 이게 문제라고 느꼈는가

> "뉴스나왔지만 미반응? 그 섹션 제거. 용도를 모르겠음. 핵심뉴스와 알고리즘 검토해서 핵심뉴스 쪽으로 병합"

이 섹션이 **어떤 투자 판단에 도움이 되는지 설명할 수 없었다.** "뉴스가 나왔는데 주가가 안 움직였다"는 정보가 투자자에게 주는 액션은 뭐지? 매수? 매도? 관망? 명확하지 않다.

### 내 판단

기능을 만들었다고 유지해야 하는 건 아니다. **"이 기능의 존재 이유를 한 문장으로 설명할 수 없으면 삭제"**가 내 기준이다. 핵심 뉴스 섹션에 해당 알고리즘의 좋은 부분만 병합하고, 독립 섹션은 제거.

### 내가 배운 것

**기능 삭제도 기획이다.** "만들어놓은 거 아까우니까 두자"는 사용자에게 노이즈를 준다. PO의 역할에는 "무엇을 빼는가"도 포함된다. 뺄 수 있는 용기가 제품을 깔끔하게 만든다.

---

## Case 13. "억대는 되어야 고래지"

### 내가 관찰한 것

"거래소 대량체결 감지" 섹션에서 소액 거래도 "고래 알림"으로 뜨고 있었다.

### 내 사고 과정

> "고래알림의 금액 상향. 억대는 되어야 고래지"

암호화폐 시장에서 "고래"란 최소 억 단위 거래를 하는 플레이어를 말한다. 천만 원대 거래를 "고래"로 분류하면 알림이 범람하고, 정작 진짜 고래 움직임이 노이즈에 묻힌다.

### 내 판단

도메인 전문 지식에 기반한 기준 재정의:
- 기존: 일정 금액 이상이면 모두 알림
- 변경: 억 단위 이상만 "고래"로 분류

기술적으로는 단순히 threshold 숫자 하나를 바꾸는 것이지만, **그 숫자를 정하는 건 코드가 아니라 도메인 이해**다.

### 내가 배운 것

**임계값(threshold) 설정은 기술이 아니라 비즈니스 판단이다.** 개발팀이 "1000만 원으로 할까요?"라고 물으면, 그건 PO가 시장에 대한 이해를 바탕으로 답해야 한다.

---

## Case 14. "코인 가격 어떻게 받아오고 있어? 이미지도 종종 안 돼"

### 내가 관찰한 것

코인 가격이 간헐적으로 안 나오고, 코인 로고 이미지도 깨지는 경우가 있었다.

### 내 사고 과정

> "코인 관련정보 잘 못 받아오는 것 같은데, 지금 가격 어떻게 받아오고 있어?"

먼저 **현재 구조를 파악**하려 했다. 문제를 고치기 전에 "지금 어떻게 되어 있는지"를 이해하는 것이 순서다.

> "CoinGecko나 CMC는 API 키 같은 거 없어? 혹은 그런 서비스 무료인데 좋은 거 없어? Web3 쪽은 꽤 있을 것 같아서"

무료 대안을 적극적으로 탐색했다. Web3 생태계의 특성상 오픈 API가 많을 거라는 가설.

> "일단 내가 외부라서 키 발급받고 그런 부분은 어렵고, 바이낸스나 CoinPaprika 이런 거 감안해서 가장 최적화된 (성능 위주, 안정성 위주) 걸 구현해줘"

제약 조건(키 발급 어려움)을 명확히 하고, 그 안에서 **성능과 안정성 기준**으로 판단을 위임했다.

### 내가 배운 것

**제약 조건을 먼저 명확히 하면 선택지가 좁아지고 결정이 빨라진다.** "최고의 코인 API를 찾아줘"가 아니라 "키 발급 없이 + 성능 위주 + 안정성 위주"로 조건을 걸면, 답이 바이낸스 WebSocket + CoinPaprika fallback으로 수렴한다.

---

## Case 15. "이유분석중만 계속 나와"

### 내가 관찰한 것

"이유 있는 움직임" 카드에 "이유분석중..."이라는 로딩 텍스트만 영구적으로 표시되고, 실제 이유는 나오지 않았다.

### 왜 이게 문제라고 느꼈는가

> "종목 보이는 카드에 이유분석중이라고 써있는데, 뭔가 노출 정보 없이 계속 이유분석중이야"

"분석 중"이라는 말은 **곧 결과가 나온다는 약속**이다. 하지만 결과가 영원히 안 나오면, 그건 거짓 약속이 된다. 사용자 신뢰를 깎는 가장 나쁜 패턴.

### 내 판단

로딩 상태가 일정 시간 이상 지속되면:
- "분석 중" → "데이터 확인 불가" 또는 해당 카드를 아예 숨기기
- **데이터가 없으면 "없다"고 솔직하게 표시**하는 것이 "곧 나옵니다"라고 거짓말하는 것보다 낫다

### 내가 배운 것

**로딩 상태에는 타임아웃이 필수다.** "분석 중"을 무한히 보여주는 건, 빈 화면보다 사용자 경험이 나쁘다. 기대를 만들어놓고 충족하지 않기 때문이다.

---

## Case 16. "섹터로테이션이 조금은 보여야 펼치기를 해볼 것 같아"

### 내가 관찰한 것

섹터 로테이션 섹션을 접힘(collapse) 처리했는데, 접힌 상태에서는 존재 자체를 모른다.

### 내 사고 과정

> "섹터로테이션이 조금은 보여야 펼치기를 해볼 것 같아"
> "섹터 로테이션 같은 건 차라리 별도 메뉴로 빼고, 핫/콜드 일부만 노출해서 별도 메뉴로 유도하면 좋잖아?"

완전히 숨기면 → 아무도 안 씀 → 없는 기능.
전부 보여주면 → 홈이 너무 길어짐.
**일부만 노출(핫/콜드 2~3개) + "더보기"** → 관심 유발 + 깊은 탐색 유도.

### 내 판단

이건 **정보 계층(Information Hierarchy)** 문제다:
- 1수준: 항상 노출 (마켓펄스, 관심종목)
- 2수준: 요약만 노출, 펼침 가능 (섹터 로테이션 핫/콜드)
- 3수준: 별도 페이지 (전체 섹터 상세)

홈 화면의 역할은 "모든 것을 보여주기"가 아니라 **"더 보고 싶게 만들기"**다.

### 내가 배운 것

**"접힘"과 "미리보기"는 다르다.** 접힌 상태 = 숨김 = 발견 불가. 미리보기 = 맛보기 = 탐색 동기 부여. UX에서 "숨기되 존재감은 유지"하는 건 세심한 설계가 필요하다.

---

## Case 17. "왜 내가 피드백한 것들 다 반영 안 됐어?"

### 내가 관찰한 것

여러 피드백을 줬는데, 배포된 버전에 관심종목 섹션 크기, 섹터 노출 방식 등이 반영되지 않았다.

### 왜 이게 문제라고 느꼈는가

> "배포된 거 맞아? 왜 내가 피드백했던 것들 다 반영 안 됐어? 예를 들어 관심종목 섹션 크기, 섹터 노출 방식, 등"

피드백을 주는 건 시간과 에너지가 든다. 그걸 했는데 반영이 안 되면 **"내 피드백이 무시됐다"는 느낌**을 받게 된다. 이건 제품 문제가 아니라 **실행력 관리** 문제다.

### 내 판단

개발팀이 피드백을 놓친 건지, 우선순위에서 밀린 건지 구분해야 한다:
- 놓친 거라면 → 피드백 트래킹 체계가 필요
- 우선순위에서 밀린 거라면 → 그 판단 근거를 공유해야 함

**PO의 피드백은 "참고 사항"이 아니라 "실행 항목"이다.** 반영하지 않을 거면 왜 안 하는지 커뮤니케이션이라도 있어야 한다.

### 내가 배운 것

**피드백 → 실행 사이의 갭을 관리하는 것도 PO의 일이다.** 피드백을 주고 "알아서 되겠지" 하면 안 된다. 트래킹하고, 확인하고, 배포 전에 체크리스트로 검증해야 한다.

---

## Case 18. "다크모드 필요성은 잘 모르겠어"

### 내가 관찰한 것

스프린트 계획에 "다크모드 지원"이 포함되어 있었다.

### 내 판단

> "일단 니 판단으로 진행. 다크모드 필요성은 잘 모르겠어"

다크모드는 "있으면 좋은" 기능이지 "없으면 서비스를 못 쓰는" 기능이 아니다. 지금 이 서비스의 우선순위는:
1. 데이터 정확성 (changePct=0 같은 버그)
2. 핵심 UX (레이아웃, 정보 계층)
3. 꾸미기 (다크모드, 애니메이션)

3번을 하기 전에 1, 2가 완벽해야 한다. **기능의 "좋음"이 아니라 "지금 필요함"으로 판단해야 한다.**

### 내가 배운 것

**"No"를 말하는 것도 우선순위 결정이다.** 모든 좋은 아이디어를 다 하면 아무것도 제대로 못 한다. "잘 모르겠어"는 "하지 마"의 완곡한 표현이 아니라, **지금은 판단 근거가 부족하니 보류**하겠다는 결정이다.

## Case 19. 사용자 피드백 8건 — "개별 이슈가 아니라 구조 문제다"

### 내가 관찰한 것

직접 앱을 사용하면서 8가지 이슈를 발견했다:

1. 마켓펄스 시장지수에 '지수 값'이 없음 (변동률만 나옴)
2. 주목할 종목 — 휴장 시 코인만 노출되어야 함
3. 거래소 체결 감지 안 됨 (코인 상장 모니터링)
4. 관심종목 영역이 너무 큼
5. 휴장 시 국장/미장 급등급락 숨겨야 함
6. 섹터 로테이션 별도 메뉴로 분리 + 핫/콜드 일부만 홈 노출
7. 뉴스 클릭 시 관심종목 매칭에 쌩뚱맞은 종목 있음
8. 상단 롤링(SurgeBanner)의 가치 재검토

### 왜 이걸 8개의 버그가 아니라 하나의 구조 문제로 봤는가

이 8개를 나열하고 보니 **공통 패턴**이 보였다:
- 1, 4, 6, 8번 → 화면에 뭘 얼마나 보여줄지 (정보 밀도)
- 2, 5번 → 상황에 따라 다르게 보여줘야 함 (컨텍스트 적응)
- 3, 7번 → 데이터 간의 연결이 안 됨 (관계 매핑)

**개별적으로 고치면 8번 고치고 끝이다. 하지만 근본 원인은 하나다: 정보 계층(Information Hierarchy)이 설계되지 않았다.** 6개 섹션이 모두 같은 크기, 같은 무게로 홈에 나열되어 있었다.

### 내 판단: 우선순위 매기기

8개를 전부 한꺼번에 할 수는 없다. 내가 사용한 기준:

```
"사용자의 투자 판단을 왜곡하는가?"
  → Yes: P0 (등락률 0%, 잘못된 지수, 뉴스 매칭 오류)
  → No but 불편: P1 (휴장 빈 화면, 관심종목 크기)
  → No and 취향: P2 (배너 디자인, 섹터 메뉴 위치)
```

이 기준으로 **P0 데이터 이슈 먼저, P1 UX 이슈 다음, P2 디자인 취향은 백로그**로 분류했다.

### 내가 배운 것

**피드백은 "리스트"가 아니라 "패턴"으로 읽어야 한다.** 8개의 개별 이슈를 보면 8개의 fix가 필요하다. 3개의 패턴(정보 밀도 / 컨텍스트 적응 / 관계 매핑)으로 묶으면, 구조적 해결이 보인다. 구조를 고치면 아직 리포트되지 않은 같은 패턴의 이슈도 예방된다.

## Case 20. "왜 같은 버그가 5번이나 나와?"

### 내가 관찰한 것

"등락률이 0%로 표시된다"는 같은 버그가 5번 이상 수정되고 5번 이상 재발했다.

### 내 사고 과정

처음 1~2번은 "고쳐줘"로 넘겼다. 3번째부터 생각이 바뀌었다:

- 1차: "미장 등락률 0이야" → 수정됨 → 해결된 줄 알았음
- 2차: "또 0이야, 이번엔 다른 종목" → 수정됨 → 불안해짐
- 3차: "ETF도 0이야" → **"이거 같은 원인 아니야?"**

3번째에서 **"이건 개별 버그가 아니라 시스템 문제"**라고 판단했다.

### 왜 이렇게 판단했는가

개발팀이 매번 "해당 파일 수정 완료"라고 했지만, 같은 로직이 4개 파일에 복사-붙여넣기 되어 있었다. A를 고치면 B에서 터지고, B를 고치면 C에서 터지는 두더지 게임이었다.

내가 요구한 것:
> "개별 파일 수정 말고, 이 로직이 어디에 몇 개 있는지 먼저 전수 조사해. 그리고 한 번에 다 고쳐."

### 결과

조사 결과: 4개 파일(`us-price.js`, `market-indices.js`, `etf-prices.js`, `stocks.js`)에 동일한 비교 함수가 중복. 절대값 비교(`0.01 차이`)를 상대값 비교(`0.01% 차이`)로 4개 파일 동시 수정 → 이후 재발 없음.

### 내가 배운 것

**PO는 "고쳐졌습니다"를 그냥 받아들이면 안 된다.** 3번째 재발에서 "왜 반복되는가?"를 물어야 한다. 개발팀에게 "이번에도 같은 방식으로 고쳤어?"라고 확인하는 것만으로도 두더지 게임을 멈출 수 있다.

## Case 21. 포지셔닝 전환 — "급등 캐치"에서 "매수 판단 5분"으로

### 내가 관찰한 것

초기 3주간 fix 커밋이 67%. 기능을 추가해도 금방 깨졌다. SurgeBanner 하나에 디자인 변경만 4번.

### 내 사고 과정

왜 이렇게 불안정한가? 기능 스펙이 흔들리는 건 **"이 기능이 왜 있는지"가 불명확하기 때문**이다.

"급상승 종목을 빠르게 캐치" — 이건 우리만의 가치가 아니다. 네이버 증권 인기검색어, 키움 HTS 급등주 스크리너로도 된다.

**진짜 질문**: 사용자가 급등주를 발견한 다음에 하는 행동은 뭐지?
→ "왜 올랐지?" 검색
→ "관련 뉴스" 확인
→ "지금 들어가도 되나?" 판단

우리가 줘야 하는 건 급등 목록이 아니라 **판단에 필요한 맥락**이다.

### 내 결정

포지셔닝을 전환했다:
```
기존: "급상승 종목을 누구보다 빠르게 캐치"
전환: "매수 결정 직전 5분을 함께하는 앱"
```

이 전환의 구체적 영향:
- "이유 없는 급등 목록은 가치 없음" → 종목마다 "왜 지금?" 1문장 컨텍스트 필수
- 뉴스가 "부가 기능"에서 "핵심 무기"로 승격 → RSS 소스 10개 추가
- "데이터 정확성"이 "디자인"보다 우선 → Phase 1을 데이터 신뢰성으로 설정

### 내가 배운 것

**포지셔닝이 흔들리면 모든 기능 결정이 흔들린다.** "이 기능 넣을까 말까?"에 대한 답이 포지셔닝에서 나와야 한다. "매수 판단에 도움이 되나?"라는 질문 하나로 섹터 로테이션(Yes, 유지), 이벤트 캘린더(Yes, 유지), DEX 핫 토큰(No, 삭제) 결정이 명확해졌다.

## Case 22. "Phase 순서가 곧 전략이다"

### 내 판단 과정

구현할 것들이 많았다. 어떤 순서로 할 것인가?

```
선택지 A: UX 먼저 → 예쁘게 만들고 → 데이터 나중에
선택지 B: 데이터 먼저 → 신뢰할 수 있게 만들고 → UX 나중에
```

나는 B를 선택했다.

**이유**: 등락률이 0%로 뜨는 화면에 아무리 예쁜 UI를 입혀도, 사용자가 "이 앱 데이터 맞아?"라고 의심하는 순간 모든 게 끝이다. **투자 앱에서 데이터 신뢰 = 서비스 존재 이유.**

```
Phase 1: 데이터 신뢰성 (기반 공사)
  → changePct=0 근본 수정, 데이터 소스 fallback 강화, 신뢰도 라벨

Phase 2: UX 재구조화 (가치 재정의)
  → 홈 레이아웃 개편, "왜 지금?" 컨텍스트, 정보 계층 설계

Phase 3: 데이터 소스 확장 (차별화)
  → 코인 250+종 확장, 뉴스 RSS 10개, KIS WebSocket
```

### 내가 배운 것

**"무엇을 먼저 하느냐"가 "무엇을 하느냐"보다 중요하다.** Phase 순서를 잘못 잡으면, Phase 2에서 만든 UI가 Phase 1에서 깨진다. "기반 공사 → 구조 → 외장" 순서는 건축에서도 소프트웨어에서도 같다.

## Case 23. "KRX 키도 있는데 이거 폴백용으로 붙여넣을까?"

### 내가 관찰한 것

한투 API만으로는 전종목 커버가 어려운 상황이었다.

### 내 사고 과정

> "맞다 KRX 키도 있는데 이거 폴백용으로 (용도나 상황은 니가 필요한 곳에) 사용하게 붙여넣을까?"

개발팀이 요청하기 전에 **내가 먼저 자원을 제공**한 케이스다. KRX API 키를 가지고 있다는 걸 떠올리고, 이게 fallback으로 쓸 수 있겠다는 판단을 했다. 용도 판단은 개발팀에 위임하되, 자원 자체는 선제적으로 제공.

확인해보니 KRX API에서 기대한 실시간 시세 기능은 없었다. 즉시 목표를 축소:

> "그런 내용 없어, 그냥 종목 리스트 확보로 가자. ETF나"

**"없으면 없는 대로 쓸 수 있는 범위를 빠르게 재정의"**하는 판단.

### 내가 배운 것

**PO가 자원(API 키, 데이터 소스)을 먼저 파악하고 제공하면 개발 속도가 빨라진다.** "키가 필요합니다" → "발급해주세요" → "발급했습니다" → "넣었습니다" 4단계가, 내가 먼저 제공하면 1단계로 줄어든다.

## Case 24. "이 프로젝트는 왜 GitHub에 정보가 제대로 안 되어 있지?"

### 내가 관찰한 것

다른 프로젝트의 GitHub 페이지를 보다가, 우리 레포에는 README, 보안 지침, 프로젝트 설명이 부실하다는 걸 발견했다.

### 왜 이게 문제라고 느꼈는가

> "왜 이 프로젝트는 깃에 이 캡쳐같이 정보 요약이나 README라던지 보안 지침 등이 제대로 안 되어 있지?"

포트폴리오로도 쓸 건데, GitHub 레포가 날것 그대로면 외부에서 볼 때 **"관리가 안 되는 프로젝트"**로 보인다. 코드 품질과 별개로, 프로젝트의 **외부 인식** 관리도 필요하다.

### 내가 배운 것

**코드만 좋아서는 안 된다. 코드를 설명하는 문서도 제품의 일부다.** 채용 담당자가 GitHub을 볼 때 코드를 한 줄 한 줄 읽지 않는다. README, 커밋 히스토리, Issue/PR 관리 상태를 본다.

## Case 25. "QA 돌려주고, 그다음 스프린트 계획 잡고, 나랑 진행은 논의"

### 내 사고 과정

> "QA 돌려주고 그다음 스프린트 진행, 계획 먼저 잡고 나랑 진행은 논의"

이건 **의사결정 프로세스 자체에 대한 결정**이다:
1. QA 먼저 (현재 품질 확인)
2. 계획 수립 (다음 할 일 정리)
3. 나와 논의 (실행 전 합의)

"알아서 해줘"가 아니라 **"계획까지는 니가 해, 실행은 내가 보고 판단할게"**라는 위임의 범위 설정.

### 내가 배운 것

**위임은 "전부 맡기기"가 아니라 "어디까지 맡기는지 경계를 정하는 것"이다.** 조사·계획은 위임하되, 실행 승인은 직접 하는 구조가 가장 효율적이면서 통제력을 유지한다.
