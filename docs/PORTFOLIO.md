## 1. 프로젝트 개요

**Market Radar**는 국내주식·미국주식·암호화폐 3개 시장의 실시간 시세를 하나의 화면에서 모니터링하는 웹 대시보드다.

| 항목 | 내용 |
|------|------|
| 기술 스택 | React + Vite, TailwindCSS, React Query, Recharts, lightweight-charts |
| 배포 | Vercel Edge Functions (서버사이드 프록시) |
| 데이터 소스 | 한투 OpenAPI, Yahoo Finance v8, Upbit WebSocket, Stooq, CoinGecko, CoinPaprika |
| 타겟 유저 | 아침 출근길 5분 안에 시장을 파악하려는 직장인 단타/스윙 트레이더 (25~40세) |

## 2. 문제 인식과 포지셔닝 전환

### 문제 발견

프로젝트 초기 3주간 커밋 18개 중 12개(67%)가 `fix:` 였다. 같은 영역에서 반복 패치가 발생했다:

| 반복 패턴 | 발생 횟수 | 내가 내린 진단 |
|-----------|-----------|---------------|
| Mock 데이터 초기 노출 (가짜 급등) | 3회 | "초기값=0" 규칙 부재 |
| 뉴스에 언론사명 그대로 노출 | 3회 | cleanTitle 패턴 불완전 |
| 관련 종목 누락 | 반복 | 하드코딩 구조의 한계 |
| SurgeBanner 디자인 변경 | 4회 | 명세 없이 만들어짐 |

**이 패턴을 보고 느낀 것**: 문제는 코드가 아니라 **설계 전제**에 있었다. 기존 전제는 "급등 종목을 보여주기"였고, 사용자가 실제로 원하는 것은 "왜 올랐는지, 아직 기회가 있는지 판단하기"였다.

### 포지셔닝 전환 결정

```
기존: "급상승 종목을 누구보다 빠르게 캐치"
     → 급등 목록을 나열하는 데이터 피드

전환: "매수 결정 직전 5분을 함께하는 앱"
     → 판단에 필요한 맥락(이유+시그널)을 제공하는 의사결정 도구
```

**왜 이렇게 전환했는가:**
- "급등 목록"은 네이버 증권도 할 수 있다. 차별화 불가.
- 사용자가 급등 목록을 본 다음 하는 행동은 "왜?"를 검색하는 것이다.
- 그 "왜?"를 같은 화면에서 즉시 제공하면, 사용자의 앱 전환이 사라진다.

**이 결정의 파급 효과:**
- 모든 기능 결정의 기준이 "이 기능이 매수 판단에 도움이 되는가?"로 통일됨
- 뉴스가 "부가 기능"에서 "핵심 무기"로 승격
- "이유 없는 급등 목록은 가치 없음"이 팀 원칙이 됨

## 3. 사용자 정의와 JTBD 설계

### Primary Persona

> "아침 8:50 지하철에서 장 준비하는 직장인 단타/스윙 트레이더 (25~40세)"

**KPI 설정 근거**: 출퇴근 환경(한 손, 흔들리는 지하철, 1~2분 집중) → 앱 열고 첫 액션까지 8초, 전체 플로우 30초 이내 완결.

### Jobs to be Done (5가지)

JTBD를 설계할 때 핵심은 "기능 목록"이 아니라 "사용자가 해결하고 싶은 상황"에서 출발하는 것이었다:

| Job | 사용자 상황 | 고통 포인트 | 우리의 해결책 |
|-----|------------|------------|-------------|
| **아침 5분 시장 파악** | 출퇴근 중, 커피 한잔 | 여러 앱 전환 없이 3개 시장 한눈에 | 홈 대시보드 통합 |
| **급등 10초 먼저 캐치** | 장중, 빠른 판단 필요 | HTS 검색하다가 이미 늦음 | SurgeBanner + 급등 스포트라이트 |
| **내 포트폴리오 뉴스 확인** | 뉴스 발생 직후, 포지션 판단 | 뉴스와 내 종목 연결이 안 됨 | 관심종목 × 뉴스 매칭 |
| **고래 동향 따라가기** | 큰 거래 포착, 방향성 힌트 | 고래 이벤트 → 종목 정보 연결 없음 | WhalePanel → ChartSidePanel |
| **섹터 로테이션 파악** | 매크로 투자자, 자금 흐름 | 국장/미장/코인 각각 봐야 함 | 3시장 통합 섹터 로테이션 |

**학습 포인트**: Job 3를 설계할 때 처음에는 "뉴스 목록"만 만들었다. 하지만 사용자 관점에서 보면 "뉴스 → 내 종목에 영향?" 플로우가 핵심이었다. 이 인사이트로 관심종목과 뉴스를 교차 매칭하는 "내 종목 뉴스" 섹션이 탄생했다.

## 4. 데이터 신뢰성 — "보이지 않는 버그"를 시스템으로 잡다

### 문제 정의

투자 앱에서 데이터 신뢰성은 곧 제품 신뢰성이다. 사용자가 "등락률 0%"를 보면 "데이터가 아직 안 왔구나"가 아니라 **"이 종목은 변동이 없구나"**로 해석한다. 이것이 오판을 유발한다.

### Case Study: changePct=0 반복 버그

이 버그는 5번 이상 수정했지만 계속 재발했다. 내가 주목한 것은 "왜 같은 버그가 반복되는가?"였다.

**1차 분석 — 증상만 수정 (실패)**
```
증상: 미장 종목 등락률이 0%로 표시
수정: previousClose 값을 직접 대입 → 일시적 해결
재발: 다른 종목에서 같은 현상
```

**2차 분석 — 패턴 발견**
```
관찰: 4개 파일(us-price, market-indices, etf-prices, stocks.js)에
      동일한 전일 종가 비교 로직이 중복 존재
가설: 절대값 비교(0.01 차이)가 소수점 자릿수가 다른 종목에서 실패
검증: 페니 주식(가격 $0.05)은 0.01 차이가 20% 변동, 지수(21,000)는 0.01이 무의미
```

**3차 분석 — 근본 원인 특정 및 시스템 수정**
```
근본 원인: almostEqual 함수가 절대값 비교 → 상대값 비교로 전환 필요
수정: Math.abs(a - b) / Math.max(Math.abs(a), Math.abs(b), 1) < 0.0001
범위: 4개 파일 동시 수정 (한 곳만 고치면 다른 곳에서 재발)
예방: 빈 결과 반환 시 dataErrors 상태 설정 → UI에서 에러 표시
```

**이 경험에서 얻은 원칙:**
> "같은 버그가 3번 나오면, 코드가 아니라 설계가 잘못된 것이다."

### Case Study: NDX 티커 오류

프로덕션에서 NASDAQ 지수가 이상하게 표시된다는 문제를 발견했다.

```
현상: NDX 지수가 18,000대로 표시 (실제 NASDAQ-100은 21,000대)
진단 과정:
  1. "혹시 데이터 소스 지연인가?" → API 직접 호출 → 18,000대 반환 → 소스 문제 아님
  2. "혹시 다른 지수인가?" → ^IXIC는 NASDAQ Composite(18,000대), ^NDX가 NASDAQ-100(21,000대)
  3. 근본 원인: 코드에서 ^IXIC(Composite)를 호출하면서 UI에 "NDX"(NASDAQ-100)로 표시
영향: 사용자가 잘못된 지수를 보고 시장 판단 → 투자 앱에서 치명적

내가 배운 것: 투자 데이터에서는 "대충 맞는" 것이 "완전히 틀린" 것보다 위험하다.
             18,000과 21,000은 둘 다 "만 단위"라 언뜻 정상으로 보이기 때문이다.
```

## 5. 정보 계층 재설계 — "많이 보여주기"에서 "빠르게 판단하기"로

### 문제 인식

사용자 피드백 8건을 종합 분석한 결과, 핵심 패턴이 보였다:

| 피드백 | 내가 읽은 본질 |
|--------|---------------|
| "관심종목 영역이 너무 큼" | 정보 계층이 평등 — 중요도 구분 없음 |
| "시장지수에 지수 값도 표시해" | 사용자가 원하는 건 % 아니라 맥락(절대값+상대값) |
| "휴장 시 급등급락 숨겨야" | 상태에 따른 적응형 UI 부재 |
| "섹터 로테이션 별도 메뉴로" | 홈에 모든 걸 넣으면 아무것도 안 보임 |
| "SurgeBanner 가치 재검토" | 화면 상단 1등 자리를 차지할 자격이 있는가? |
| "뉴스 관련종목 매칭 이상" | 데이터 연결성 부족 |

**내 진단:** 문제는 개별 기능이 아니라 **정보 계층(Information Hierarchy)** 자체였다. 6개 섹션이 모두 같은 시각적 무게를 가지고 있어서, 사용자가 "어디를 먼저 봐야 하는지" 판단할 수 없었다.

### 해결: 3단계 정보 계층

```
Tier 1 — 즉시 인지 (0~3초): 핵심 시그널 카드 + 시장 지수
  → "지금 시장에서 가장 중요한 것 1가지"

Tier 2 — 탐색 (3~15초): 관심종목 현황 + 급등/급락 통합
  → "내 포트폴리오 상태 + 시장 전체 움직임"

Tier 3 — 심화 (15초+): 뉴스 피드 + 섹터 로테이션
  → "더 깊이 알고 싶으면 여기"
```

### 적용: 휴장 시 UX

**기존**: 장이 닫히면 급등/급락 섹션이 빈 오버레이로 대체 → 화면의 1/3이 "죽은 공간"

**내 판단**: 휴장이어도 "전일 종가 기준 데이터"는 가치가 있다. 사용자가 "어제 뭐가 올랐지?"를 확인하고 싶을 수 있다.

**결정**: 오버레이 제거 → 딤 처리(opacity 60%) + 작은 라벨("🌙 장마감 · 전일 종가 기준")
- 데이터는 유지하되, "이건 실시간이 아니다"를 명확히 표시
- 화면 활용도 유지 + 데이터 신뢰성 투명성 확보

## 6. 피드백 루프와 의사결정 과정

### 피드백 → 팀 회의 → 액션 아이템 → 배포

사용자 피드백 8건을 팀 회의 안건으로 가져왔을 때, 6명이 각자 관점에서 분석했다:

| 역할 | 관점 | 피드백 "관심종목 영역이 너무 큼"에 대한 의견 |
|------|------|-------------------------------------------|
| CPO | KPI 기반 | "스크롤 뎁스 데이터 없이 판단 불가. 하지만 첫 화면에 6개 섹션은 과하다" |
| Strategy | 사용자 가치 | "관심종목은 개인화의 핵심. 줄이되 삭제는 안 됨" |
| Designer | 정보 설계 | "높이가 문제가 아니라 '접기' 기능 부재가 문제" |
| FE | 구현 영향 | "접기 구현 간단하지만, 상태 저장(localStorage)까지 고려해야 함" |
| BE | 데이터 영향 | "관심종목 30개 이상이면 폴링 부하 — 10초→30초 조정 필요" |
| QA | 엣지 케이스 | "관심종목 0개일 때 빈 상태 UX는?" |

**내 최종 결정**: 높이 축소(5행→3행) + "더보기" 펼침 + 빈 상태 온보딩 UI

이 과정에서 배운 것:
> PO의 역할은 "모든 의견을 수용하는 것"이 아니라, 의견 충돌 시 사용자 가치 기준으로 우선순위를 매기는 것이다.

### 우선순위 결정 프레임워크

모든 피드백/버그를 처리할 때 사용한 기준:

```
P0 (즉시): 사용자가 잘못된 판단을 내릴 수 있는 데이터 오류
P1 (이번 주): 사용자 경험을 방해하지만 오판을 유발하지는 않는 문제
P2 (백로그): 있으면 좋지만 없어도 핵심 가치에 영향 없는 개선
```

**적용 예시:**
- changePct=0 (등락률 오표시) → **P0**: 투자 판단에 직접 영향
- NDX 티커 오류 (다른 지수 표시) → **P0**: 잘못된 시장 정보
- 휴장 시 빈 화면 → **P1**: 불편하지만 오판 유발 아님
- SurgeBanner 디자인 → **P2**: 취향 영역

## 7. 보안과 품질 — PO가 왜 SSRF를 신경 써야 하는가

### CTO급 감사 결정 배경

단순히 "보안 점검하자"가 아니라, **왜 지금 해야 하는지**에 대한 판단이 있었다:

1. Vercel Edge Functions이 외부 URL을 받아 서버에서 fetch하는 구조 = SSRF 위험 구조
2. 우리 앱은 투자 관련 → 보안 사고 시 신뢰 회복 불가능
3. 프로덕션 배포 전 보안 감사는 "비용"이 아니라 "보험"

### 발견된 취약점 3건

| 파일 | 취약점 | 위험 | 수정 |
|------|--------|------|------|
| api/rss.js | ALLOWED_DOMAINS 정의만 하고 검증 안 함 | 임의 URL 프록시 가능 | 도메인 검증 로직 추가 |
| api/news-summary.js | 사용자 URL을 검증 없이 크롤 | 내부 네트워크 접근 가능 | 프로토콜+내부IP 차단 |
| api/chart-proxy.js | 파라미터 화이트리스트 없음 | URL 인젝션 가능 | range/interval/symbol 검증 |

**PO 관점에서 이 발견의 의미:**

> 기능이 100% 완벽해도, 보안 취약점 하나로 서비스 전체의 신뢰가 무너진다.
> 배포 전 QA가 "SSRF 미수정 상태에서 프로덕션 배포 불가"라고 판단했을 때,
> 기능 출시를 미루고 보안 수정을 우선하는 결정을 내렸다.

## 8. 운영 체계와 팀 리더십

### 6인 에이전트 팀 운영

| 역할 | 담당 | 핵심 원칙 |
|------|------|-----------|
| CPO | 이준혁 | "숫자 없는 논의는 논의가 아니에요" |
| Strategy | 이지원 | "사용자에게 어떤 가치인가" |
| Designer | 최유나 | "감정 설계가 핵심이에요" |
| FE Lead | 박서연 | "성능이 곧 UX예요" |
| BE Lead | 김민준 | "fallback 없는 API는 API가 아니에요" |
| QA Lead | 장성민 | "이 케이스는 생각해봤어요?" |

### 운영 루틴

```
매일: 스탠드업 (3줄 보고 + 블로커 해소)
매 PR: 봇 3인 리뷰 → 종합 코멘트 → 토론 1회전 → 최종 판단 → 머지
매주: 스프린트 플래닝 + 회고
이슈 발생 시: 팀 회의 소집 → 다관점 분석 → 합의 → 실행
```

### ADR (Architecture Decision Record) 문화

모든 중요 결정을 ADR로 기록했다. 이유:
- 3주 후에 "왜 이렇게 했지?" 추적 가능
- 팀원 교체 시 컨텍스트 전달
- 잘못된 결정도 기록 → 같은 실수 방지

**ADR 예시 — 색상 시스템:**
```
결정: Coinbase Design System 구조 차용 + 색상만 한국 컨벤션 오버라이드
근거: CDS는 국제 기준(green=상승). 한국 투자자에게 빨강=상승이 체화됨.
      색상 변경 시 신뢰 깨짐. 구조는 우수하므로 그대로 차용.
리스크: CDS 업데이트 시 색상 오버라이드 깨질 수 있음 → 토큰 레이어에서 분리
```

## 10. 배운 것과 다음 단계

### PO/PM으로서 배운 것

1. **"같은 버그 3번 = 설계 결함"**: 증상이 아니라 구조를 고쳐야 한다. changePct=0이 5번 재발한 건 코드 버그가 아니라 "절대값 비교"라는 설계 전제가 잘못됐기 때문이었다.

2. **"데이터 앱에서 '대충 맞는 것'이 '완전히 틀린 것'보다 위험하다"**: NDX 지수가 18,000(Composite)으로 표시됐을 때, 대부분의 사용자는 "뭔가 이상한데?"조차 느끼지 못한다. 21,000(실제 NDX)과 크게 다르지 않아 보이니까. 이런 "거의 맞는 오류"가 투자 판단을 가장 크게 왜곡한다.

3. **"보안은 기능 출시보다 우선"**: SSRF 취약점을 발견했을 때, 새 기능 배포를 멈추고 보안 수정을 먼저 했다. 투자 앱에서 보안 사고는 신뢰 회복이 불가능하기 때문이다.

4. **"피드백은 표면이 아니라 본질을 읽어야 한다"**: "관심종목 영역이 너무 큼"은 높이 문제가 아니라 정보 계층 부재의 증상이었다. 높이만 줄이면 다른 곳에서 같은 불만이 나온다.

5. **"Phase 순서가 곧 전략"**: Phase 1(데이터 신뢰성) → Phase 2(UX 재구조화) → Phase 3(데이터 소스 확장) 순서를 지킨 이유는 "신뢰할 수 없는 데이터 위에 아무리 좋은 UX를 올려도 의미 없다"는 판단이었다.
