# /스크럼 명령어

## 사용법
```
/스크럼 [총시간] [주기]
/스크럼 3h 30m   → 3시간 동안, 30분마다
/스크럼 2h 20m   → 2시간 동안, 20분마다
/스크럼 1h 15m   → 1시간 동안, 15분마다
```

인수($ARGUMENTS): $ARGUMENTS

---

## 실행 지침

아래를 순서대로 실행하라.

### 0. 인수 파싱

`$ARGUMENTS`에서 총시간(시간)과 주기(분)를 파싱한다.
- 예: "3h 30m" → total_hours=3, interval_minutes=30
- 예: "2h 20m" → total_hours=2, interval_minutes=20
- 인수가 없으면 기본값: total_hours=2, interval_minutes=30

파싱한 값으로 `.claude/scrum_session.json`을 생성 또는 업데이트한다:

```json
{
  "started_at": "<현재 ISO 타임스탬프>",
  "total_minutes": <total_hours * 60>,
  "interval_minutes": <interval_minutes>,
  "iteration": 1,
  "status": "running",
  "last_run": "<현재 ISO 타임스탬프>",
  "deployed_at": null,
  "pending_deploy": false
}
```

### 1. 현황 파악 (6 에이전트 병렬)

다음 파일들을 읽고 각 에이전트 관점에서 현황을 분석한다:
- `git log --oneline -10` → 최근 커밋
- `git diff --stat HEAD~3..HEAD` → 최근 변경사항
- `.project/strategy.md` → 전략/JTBD 현황
- `.project/tech-debt.md` → 기술 부채
- `.claude/scrum_session.json` → 이전 세션 상태
- `src/api/`, `src/components/` → 코드 현황

**이지원(Strategy):** JTBD 5개 대비 현재 gap, 사용자에게 제공 못 하는 본질 가치
**최유나(Design):** 정보 계층, 색상 토큰 일관성, 모바일 UX 위험
**이준혁(PM):** 완성 기능 vs 미완성 기능, 비즈니스 우선순위
**김민준(BE):** API fallback 체인, rate limit, 데이터 freshness
**박서연(FE):** 죽은 코드, 성능 위험, 컴포넌트 구조 문제
**장성민(QA):** 에러 처리 경로, 코인 급등 시나리오 취약점

### 2. 서비스 가치·우선순위 논의 (이준혁 × 이지원 × 최유나)

세 명이 토론한다. 주제:
- 지금 사용자가 앱을 켜는 이유가 뭔가? (JTBD 관점)
- 그 이유를 현재 화면이 5초 안에 충족시키는가?
- 충족 못 하는 가장 큰 이유 한 가지
- 다음 작업 우선순위 P0/P1/P2 재산정

논의 결과를 `.project/strategy.md`에 반영한다.

### 3. 에이전트별 문제·개선사항 토론

나머지 3명(김민준, 박서연, 장성민)이 발언:
- 발견한 버그/개선사항을 각자 P0/P1/P2로 분류
- 6명 합의 → 최종 작업 목록 확정

할일이 없거나 P0가 없으면:
```
## 📋 대표님 보고 요청
현재 P0 이슈 없음. 서비스가 안정적으로 동작 중.
논의 필요 사항: [내용]
회의 요청: [질문 or 방향성 확인 필요 항목]
```
을 출력하고 종료한다.

### 4. 작업 실행

P0 → P1 → P2 순서로 즉시 코드 수정:
- 파일 읽기 → 수정 → 빌드 확인(`npm run build`)
- 각 작업 완료 후 `.project/tech-debt.md` 업데이트

**큰 작업 단위 완료 기준** (아래 중 하나):
- P0 버그 수정 완료
- 새 기능 구현 완료 (PRD 기능 하나)
- API fallback 체인 개선 완료

### 5. 큰 작업 완료 시 → 배포 + 대표님 노티

큰 작업이 완료되면:

```bash
git checkout -b auto/scrum-$(date +%Y%m%d-%H%M)
git add -A
git commit -m "auto: [작업 내용 한줄 요약]"
git push origin [브랜치명]
gh pr create --title "[작업 내용]" --body "[변경사항 요약]"
gh pr merge [PR번호] --squash --admin --subject "[작업 내용]"
git checkout main && git reset --hard origin/main
vercel --prod --yes
```

배포 완료 후 반드시 아래 형식으로 출력한다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚀 배포 완료 — 대표님 확인 요청
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
URL: https://market-dashboard-v2-mu.vercel.app
완료 작업: [내용]
변경 파일: [목록]
다음 계획: [P1 작업 or 새 기능 시작]

⏱ 잔여 세션: [남은 시간]분
💬 계속 진행할까요? 새 지침이 있으면 말씀해 주세요.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

그 후 `.claude/scrum_session.json`의 `pending_deploy: false`, `deployed_at: <현재>`로 업데이트.

### 6. 세션 상태 업데이트 + 회의록

`.claude/scrum_session.json`의 `iteration`을 +1, `last_run`을 현재로 업데이트.

`.project/SCRUM_LOG.md`에 이번 사이클 결과 추가.

### 7. CronCreate 등록 (첫 실행 시만)

`scrum_session.json`의 `iteration == 1`인 경우에만 CronCreate를 등록한다.
(이후 반복은 이미 등록된 크론이 처리)

크론 주기: `interval_minutes`에서 cron expression 계산
- 30분 → `*/30 * * * *`
- 20분 → `*/20 * * * *`
- 15분 → `*/15 * * * *`
- 45분 → `*/45 * * * *`

CronCreate의 prompt는 다음 텍스트를 사용:

```
마켓레이더 자동 스크럼 사이클 실행.

1. .claude/scrum_session.json 읽기
2. started_at + total_minutes로 잔여 시간 계산. 만료됐으면:
   "⏰ 스크럼 세션 종료 (총 [N]회 완료). 대표님, 세션이 종료되었습니다." 출력 후 종료.
3. 잔여 시간 있으면: /스크럼 명령어의 Step 1~6을 그대로 실행.
   - .claude/agents/ 폴더의 페르소나 파일 참고
   - .project/strategy.md, .project/tech-debt.md 참고
   - PRD_v3.md 참고
   - 큰 작업 완료 시 배포 + 대표님 노티 (Step 5 형식 그대로)
4. iteration +1, last_run 업데이트
```
