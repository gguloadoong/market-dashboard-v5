## 평가 체계

### 5개 평가 항목 (각 20점, 만점 100점)

| # | 항목 | 가중치 | 최대 가중점 | 평가 기준 |
|---|------|--------|-----------|----------|
| 1 | 기획 의도 부합 | x3 | 60 | 기획서에 명시된 기능/UX가 구현되었는가. 누락·변형·과잉 구현 감점 |
| 2 | 투자 인사이트 제공 | x2 | 40 | 사용자가 이 기능으로 투자 판단에 도움을 얻을 수 있는가 |
| 3 | 차별화 시그널 경험 | x2 | 40 | 다른 서비스에서 얻기 어려운 고유한 가치가 있는가 |
| 4 | 디자인/UX 직관성 | x2 | 40 | 첫 사용자도 3초 내 이해할 수 있는가. 정보 계층이 명확한가 |
| 5 | 정확도/커버리지 | x3 | 60 | 시세·뉴스·시그널이 정확하고, 커버리지가 충분한가 |

**총 가중 만점: 240점**

### 등급 기준

| 등급 | 가중 점수 | 비율 | 판정 |
|------|----------|------|------|
| S | 216+ | 90%+ | **PASS** — 머지 승인 |
| A | 192~215 | 80~89% | CONDITIONAL — 지적사항 수정 후 재평가 |
| B | 168~191 | 70~79% | REWORK — 주요 수정 필요 |
| C | <168 | <70% | REJECT — 재설계 필요 |

### 세부 채점 가이드 (항목별 20점 내)

```
20점: 완벽 — 개선할 것 없음
16~19: 우수 — 사소한 개선점만 존재
12~15: 양호 — 핵심은 충족, 개선 필요
8~11: 미흡 — 기본은 되나 주요 갈이 있음
4~7: 부족 — 핵심 요소 누락
0~3: 미구현 또는 완전 불일치
```

## 실행 프로세스

```
[Phase N 시작]
    │
    ▼
┌─────────────────────────┐
│ Team A: 기획팀           │
│ 기획서 작성              │
│ (기능 스펙 + UI + 기준)  │
└────────┬────────────────┘
         │ 기획서 전달
         ▼
┌─────────────────────────┐
│ Team B: 구현팀           │
│ 이슈 생성 → 브랜치       │
│ 구현 → 빌드 → 코드리뷰   │
│ PR 생성                  │
└────────┬────────────────┘
         │ PR + 빌드 결과
         ▼
┌─────────────────────────┐
│ Team C: 평가팀           │
│ 5개 항목 정량 평가       │
│ 점수 산출 + 개선사항      │
└────────┬────────────────┘
         │
         ▼
    90%+ 달성?
     ├─ YES → 머지 → 다음 Phase
     └─ NO → 개선사항 → Team B로 돌아감 (반복)
```

### 반복 규칙

1. 평가 결과 90% 미만 → 개선사항 목록을 Team B에 전달
2. Team B는 개선사항만 수정 (새 기능 추가 금지)
3. 수정 후 다시 Team C 평가
4. 최대 3회 반복 — 3회 후에도 90% 미달 시 Team A가 기획 재검토
5. 각 반복은 이전 평가의 감점 항목만 재평가 (전체 재평가 아님)
